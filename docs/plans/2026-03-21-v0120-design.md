# cxpak v0.12.0 Design: Data Layer Awareness

## Summary

Make cxpak understand the entire data layer of a codebase ÔÇö SQL schemas, ORM models, infrastructure-as-code database definitions, embedded SQL in application code, and migration ordering. A new `src/schema/` module detects, extracts, and links data layer definitions regardless of what database technology the project uses. The dependency graph gains typed edges so every relationship carries semantic meaning.

## Architecture

New module: `src/schema/`
- `mod.rs` ÔÇö public API, types (`SchemaIndex`, `TableSchema`, `ColumnSchema`, etc.)
- `extract.rs` ÔÇö column-level SQL DDL extraction, CQL, Cypher, Elasticsearch
- `detect.rs` ÔÇö post-parse ORM pattern matching, Terraform tagging, migration ordering
- `link.rs` ÔÇö embedded SQL detection, ORM model Ôćĺ table linking, schema edge production

Modified module: `src/index/graph.rs`
- `DependencyGraph` edges gain `EdgeType` (full migration of all consumers)
- `TypedEdge` struct replaces raw `String` in edge sets

The schema module runs post-parse, reads the already-built `CodebaseIndex`, and produces a `SchemaIndex` that is stored alongside it. Schema edges are added to the unified `DependencyGraph`.

### SQL Column-Level Extraction

The SQL parser's `extract()` in `src/parser/languages/sql.rs` already walks the tree-sitter AST for CREATE TABLE/FUNCTION/VIEW/INDEX/TRIGGER. The schema extraction layer (`src/schema/extract.rs`) takes the parsed symbols and performs deeper extraction:

```rust
pub fn extract_sql_schema(file: &IndexedFile) -> Vec<SchemaEntry> {
    // Walk parsed symbols
    // For each CREATE TABLE symbol:
    //   - Re-parse the body to extract column definitions
    //   - Extract column name, type, constraints from each column_definition node
    //   - Extract FOREIGN KEY constraints Ôćĺ ForeignKeyRef
    //   - Extract PRIMARY KEY (inline or table-level)
    // For each CREATE VIEW:
    //   - Extract table names from the SELECT body (words after FROM, JOIN)
    // For each CREATE FUNCTION:
    //   - Scan body for table references
    // For each CREATE INDEX:
    //   - Extract target table and column list
    // For each CREATE TRIGGER:
    //   - Extract target table and referenced function
}
```

Column extraction parses the symbol body (already available in `Symbol.body`). Primary approach: **regex-based extraction** from the body text. The CREATE TABLE body has a well-defined format: `(column_name TYPE [constraints], ...)`. Parse column definitions by splitting on commas at the top parenthesis level, then extracting name (first word), type (second word/phrase), and constraints (remaining words: NOT NULL, UNIQUE, PRIMARY KEY, DEFAULT ..., REFERENCES ...).

This is more reliable than tree-sitter re-parse because tree-sitter-sql (v0.0.2) is immature and its child node types for column definitions are unverified. Regex on DDL is deterministic ÔÇö the syntax is standardized and unambiguous.

FOREIGN KEY detection: match `REFERENCES table_name(column_name)` pattern in column definition or as a table-level constraint.

**Dialect handling:** Tree-sitter-sql parses the AST structure. Type strings are captured as raw text ("VARCHAR(255)", "JSONB", "SERIAL") without interpretation. Dialect-specific clauses that don't parse (ClickHouse ENGINE, Cassandra WITH) are skipped ÔÇö no errors.

**CQL (Cassandra):** `.cql` extension mapped to SQL parser. CQL CREATE TABLE syntax is SQL-compatible for column definitions. Cassandra-specific clauses treated as unparsed blocks.

**ClickHouse:** Already handled by SQL parser. ENGINE clauses after the column definitions are skipped gracefully.

### Cypher (Neo4j) Extraction

No tree-sitter crate available. Regex-based extraction from `.cypher` files:

```rust
pub fn extract_cypher_schema(content: &str) -> Vec<SchemaEntry> {
    // Regex patterns:
    // CREATE CONSTRAINT ... FOR (n:Label) REQUIRE n.property ...
    // CREATE INDEX ... FOR (n:Label) ON (n.property)
    // Node label patterns (:Label) in any context
    // Relationship patterns -[:REL_TYPE]->
}
```

Extracts: node labels, relationship types, constraints, indexes. No column-level detail (Neo4j is schemaless) but labels and constraints define the data model.

### Elasticsearch Extraction

Pattern detection on already-parsed JSON files:

```rust
pub fn extract_elasticsearch_schema(file: &IndexedFile) -> Option<SchemaEntry> {
    // Check if JSON file contains "mappings" key with "properties" sub-key
    // Extract field names and types from properties object
    // Nested properties Ôćĺ dot-notation field names ("user.address.city")
}
```

Only triggers on JSON files containing the `"mappings"` + `"properties"` pattern. Other JSON files are ignored.

## Feature 3: Schema Detection (ORM + Infrastructure)

### ORM Pattern Matchers

Post-parse pass over `CodebaseIndex`. Each matcher is a function:

```rust
pub fn detect_orm_models(index: &CodebaseIndex) -> Vec<OrmModelSchema>
```

**Detection patterns:**

| ORM | Language | Pattern | Table Name Resolution |
|---|---|---|---|
| Django | Python | Symbol kind `Class`, signature contains `models.Model` | `class Meta: db_table = "X"` in body, else lowercase class name |
| SQLAlchemy | Python | Symbol kind `Class`, signature contains `(Base)` or `(DeclarativeBase)` **AND** file imports contain `sqlalchemy` | `__tablename__ = "X"` in body, else lowercase class name |
| TypeORM | TypeScript | Symbol kind `Class`, body contains TypeORM member decorators (`@Column`, `@PrimaryColumn`, `@ManyToOne`, `@OneToMany`, etc.) | `@Entity("X")` from `file.content` (line before class), else lowercase class name |
| ActiveRecord | Ruby | Symbol kind `Class`, signature contains `< ActiveRecord::Base` or `< ApplicationRecord` | Lowercase pluralized class name: (1) lowercase, (2) if ends in `s`/`ss`/`sh`/`ch`/`x`/`z` Ôćĺ append `es`, (3) if ends in consonant+`y` Ôćĺ replace `y` with `ies`, (4) default append `s`. (`User`Ôćĺ`users`, `Category`Ôćĺ`categories`, `Address`Ôćĺ`addresses`) |
| Prisma | Prisma | Symbol kind `Struct` (already extracted by parser) | `@@map("X")` in body, else lowercase model name |

**Pluralization for ActiveRecord:** Simple rule-based: `s` Ôćĺ `ses`, `y` Ôćĺ `ies`, `x`/`ch`/`sh` Ôćĺ `es`, default append `s`. Covers 95% of cases. No inflector crate needed.

**Field extraction:** For each ORM model, extract field definitions from the body:
- Django: `name = models.CharField(...)`, `user = models.ForeignKey(User, ...)`
- SQLAlchemy: `name = Column(String)`, `user_id = Column(Integer, ForeignKey("users.id"))`
- TypeORM: `@Column() name: string`, `@ManyToOne(() => User) user: User`
- ActiveRecord: inferred from migration files (not model body)
- Prisma: `name String`, `author User @relation(...)`

## Feature 4: Schema Linking (Cross-Language)

### Embedded SQL Detection

Scan string literals in symbol bodies for SQL patterns:

```rust
const SQL_KEYWORDS: &[&str] = &[
    "SELECT ", "INSERT ", "UPDATE ", "DELETE ",
    "CREATE ", "ALTER ", "DROP ",
    "FROM ", "JOIN ", "INTO ",
];

pub fn detect_embedded_sql(body: &str) -> Vec<EmbeddedSqlRef> {
    // 1. Find string literals (content between quotes)
    // 2. Check if any contain SQL keywords
    // 3. Require structural SQL pattern: keyword + FROM/INTO/TABLE to reduce false positives
    // 4. Extract table names: word immediately after FROM, JOIN, INTO, UPDATE, TABLE
    // 5. Return (table_name, sql_fragment) pairs
}

pub struct EmbeddedSqlRef {
    pub table_name: String,
    pub sql_fragment: String,
    pub line: usize,
}
```

**False positive prevention:** A string must contain both a DML/DDL keyword (SELECT, INSERT, etc.) AND a structural keyword (FROM, INTO, TABLE) to be considered SQL. `"SELECT the best option"` is not SQL because it has no FROM. `"SELECT * FROM users"` is SQL.

**Applies to all languages.** String literal detection is language-agnostic ÔÇö scan the raw symbol body for quoted strings. Additionally, scan `file.content` directly for files with no parse result or for SQL outside symbol bodies (e.g., module-level code, top-level strings). Deduplicate: if a table reference was already found in a symbol body, don't re-add it from file content.

### ORM Model Linking

The schema detection layer resolves each ORM model to a table name. The linking step produces edges:

1. **Model file Ôćĺ schema file:** If `models/user.py::User` maps to table `users` and `schema/tables.sql` defines `CREATE TABLE users`, create `OrmModel` edge.
2. **Model file Ôćĺ model file (via relations):** Django `ForeignKey(User)` in `Order` model Ôćĺ `OrmModel` edge from order model file to user model file.

When no SQL file exists for a table (ORM defines the schema), the model IS the schema source ÔÇö tagged accordingly in `SchemaIndex`.

### Edge Production

```rust
pub fn build_schema_edges(
    index: &CodebaseIndex,
    schema_index: &SchemaIndex,
) -> Vec<(String, String, EdgeType)> {
    let mut edges = Vec::new();

    // FK edges: table file Ôćĺ referenced table file
    for (_, schema) in &schema_index.tables {
        for col in &schema.columns {
            if let Some(fk) = &col.foreign_key {
                if let Some(target) = schema_index.tables.get(&fk.target_table) {
                    edges.push((schema.file_path.clone(), target.file_path.clone(), EdgeType::ForeignKey));
                }
            }
        }
    }

    // View Ôćĺ source table edges
    for (_, view) in &schema_index.views {
        for table_name in &view.source_tables {
            if let Some(table) = schema_index.tables.get(table_name) {
                edges.push((view.file_path.clone(), table.file_path.clone(), EdgeType::ViewReference));
            }
        }
    }

    // Function Ôćĺ referenced table edges
    for (_, func) in &schema_index.functions {
        for table_name in &func.referenced_tables {
            if let Some(table) = schema_index.tables.get(table_name) {
                edges.push((func.file_path.clone(), table.file_path.clone(), EdgeType::FunctionReference));
            }
        }
    }

    // Embedded SQL edges (scan all files)
    for file in &index.files {
        if let Some(pr) = &file.parse_result {
            for symbol in &pr.symbols {
                for sql_ref in detect_embedded_sql(&symbol.body) {
                    if let Some(table) = schema_index.tables.get(&sql_ref.table_name) {
                        edges.push((file.relative_path.clone(), table.file_path.clone(), EdgeType::EmbeddedSql));
                    }
                }
            }
        }
    }

    // ORM model Ôćĺ table edges
    for (_, model) in &schema_index.orm_models {
        if let Some(table) = schema_index.tables.get(&model.table_name) {
            edges.push((model.file_path.clone(), table.file_path.clone(), EdgeType::OrmModel));
        }
    }

    // Migration sequence edges
    for chain in &schema_index.migrations {
        for i in 1..chain.migrations.len() {
            edges.push((
                chain.migrations[i].file_path.clone(),
                chain.migrations[i - 1].file_path.clone(),
                EdgeType::MigrationSequence,
            ));
        }
    }

    edges
}
```

## Integration with Existing Pipeline

### Index Build Time Flow

```
Scanner Ôćĺ Parser Ôćĺ CodebaseIndex::build()
                         Ôćô
                   SchemaExtractor (enhanced SQL column-level parsing)
                         Ôćô
                   SchemaDetector (ORM patterns, migration ordering, Terraform tagging)
                         Ôćô
                   SchemaIndex (stored on CodebaseIndex as Option<SchemaIndex>)
                         Ôćô
                   build_dependency_graph() Ôćĺ Import edges (EdgeType::Import)
                         Ôćô
                   build_schema_edges() Ôćĺ FK, View, ORM, EmbeddedSql, Migration edges
                         Ôćô
                   Unified DependencyGraph with typed edges
```

`SchemaIndex` is stored as `pub schema: Option<SchemaIndex>` on `CodebaseIndex`. `None` for repos with no schema files.

**Orchestration:** `build_dependency_graph()` is extended to accept an optional `&SchemaIndex`. When present, it calls `build_schema_edges()` internally and adds the resulting typed edges to the same graph alongside the import edges. Callers pass `index.schema.as_ref()` ÔÇö no separate call needed. This keeps the single-call pattern that all consumers already use.

### Trace

`cxpak trace users_table` finds the `users` table in `SchemaIndex`, walks FK edges to related tables, follows `EmbeddedSql` edges to code files that query it, follows `OrmModel` edges to model classes. Output includes table schema detail (columns, types, FKs) rendered using v0.11.0 detail levels.

### Pack Context

Annotations (v0.11.0) gain a `schema:` line when the file contains schema definitions:
```
// [cxpak] schema/tables.sql
// score: 0.82 | role: dependency | parent: src/api/orders.py (via: embedded_sql)
// detail_level: full (1,200 tokens)
// schema: table "orders" ÔÇö 8 columns, 2 FKs (users, products), 1 index
```

### Context for Task

Query expansion (v0.11.0) gains schema-aware expansion when `Database` domain is active AND `SchemaIndex` exists. Table and column names added as expansion terms.

## Design Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Post-parse schema detection | Layer on top of existing parsers | Parsers stay simple; schema is a separate concern |
| Separate `SchemaIndex` | Not embedded in `Symbol` | Clean isolation; most pipeline consumers don't need schema info |
| Full edge type migration | Change all 10 call sites | Every consumer benefits; doing it incrementally wastes time |
| Column-level extraction (Option B) | Deep DDL parsing, not query analysis | Deterministic ÔÇö DDL has one correct parse. DML is context-dependent. |
| String literal SQL scanning | Not API-aware | Language-agnostic, no framework treadmill, deterministic |
| Embedded SQL requires FROM/INTO/TABLE | Structural keyword check | Prevents false positives like "SELECT the best option" |
| ORM table name resolution | Convention-based with override detection | `db_table`, `__tablename__`, `@Entity("X")`, `@@map("X")` ÔÇö all deterministic |
| ActiveRecord pluralization | Simple rule-based | Covers 95% of cases without inflector dependency |
| Migration ordering | Directory pattern + filename sorting | Framework-agnostic, deterministic, no content parsing needed |
| Cypher via regex | No tree-sitter crate available | Cypher DDL is small and well-defined |
| Elasticsearch via JSON patterns | Already parsed by JSON parser | Just pattern detection on existing data |
| CQL reuses SQL parser | Near-identical DDL syntax | One parser, two extensions |
| ClickHouse handled gracefully | SQL parser skips ENGINE clause | No errors on dialect-specific syntax |
| SchemaIndex as Option | `None` when no schema files | Features degrade gracefully; no overhead for non-DB repos |
| `.pl` Ôćĺ Perl (v0.10.0 scope) | Known ambiguity with Prolog | Perl far more common; same pattern as `.m` Ôćĺ Obj-C. If Perl parser not yet shipped, `.pl` files still get embedded SQL detection via raw content scan. |
