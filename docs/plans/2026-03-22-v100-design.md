# cxpak v1.0.0 Design: One Call, Perfect Context

## Summary

The capstone release. `cxpak_auto_context` is a single MCP tool that takes a task description and returns the optimal context bundle — automatically selecting, ranking, filtering, packing, and annotating the best context from the entire codebase. Noise filtering actively excludes content that hurts LLM performance. Local embeddings add a 7th scoring signal with zero configuration. `cxpak_context_diff` provides lightweight deltas for long sessions. The v1.0.0 MCP API is stable under semver.

## Architecture

New module: `src/auto_context/`
- `mod.rs` — public API, orchestration pipeline
- `noise.rs` — blocklist, similarity dedup, relevance floor
- `briefing.rs` — output formatting, section assembly, budget allocation
- `diff.rs` — context diff snapshot + delta computation

New module: `src/embeddings/` (behind `embeddings` Cargo feature)
- `mod.rs` — public API
- `model.rs` — model download, caching, loading via candle
- `index.rs` — embedding index build, incremental update, cosine similarity search

Modified: `src/relevance/` (7th signal), `src/commands/serve.rs` (2 new MCP tools), `Cargo.toml` (candle dependencies)

```json
      },

        "type": "string",

      },

        "type": "boolean",

        "default": true
      },
      "include_blast_radius": {
```

**No `format` parameter:** Unlike `cxpak_overview` which produces markdown/JSON/XML output, `auto_context` always returns structured JSON with embedded source content. The `content` fields contain raw annotated source code (with language-aware comment annotations prepended). This is intentional — the MCP protocol transports JSON, and the LLM receives the source as-is. There's no need for markdown wrapping since the content IS source code.

### Orchestration Pipeline

```
task string
    ↓
1. Query expansion (core + domain synonyms from v0.11.0)
    ↓
2. Relevance scoring (7 signals including PageRank + embeddings)
    ↓
3. Seed selection + dependency fan-out (v0.9.0)
    ↓
4. Noise filtering (blocklist + similarity dedup + relevance floor)
    ↓
5. Test file mapping (auto-include from v0.13.0)
    ↓
6. Schema context (if database domain active, from v0.12.0)
    ↓
7. Blast radius preview (for top-scored files, from v0.13.0)
    ↓
8. API surface (public signatures of related modules, from v0.13.0)
    ↓
9. Fill-then-overflow budget allocation with progressive degradation (v0.11.0)
    ↓
10. Context annotations on every piece (v0.11.0)
    ↓
Output: structured briefing
```

Steps 1-3 and 5-8 reuse existing infrastructure. Step 4 (noise filtering) and step 9 (fill-then-overflow) are the new orchestration logic.

**Function call sketch for each step:**

```rust
pub fn auto_context(task: &str, index: &CodebaseIndex, budget: usize, opts: &AutoContextOpts) -> AutoContextResult {
    // 1. Query expansion
    let expanded = context_quality::expansion::expand_query(task, &index.domains);

    // 2. Relevance scoring (with embeddings if available)
    let scorer = MultiSignalScorer::new_for_index(index); // picks 6 or 7 signal weights
    let scored = scorer.score_all_expanded(task, &expanded, index);

    // 3. Seed selection + fan-out
    let seeds = relevance::seed::select_seeds_with_graph(&scored, index, SEED_THRESHOLD, 50, Some(&index.graph));

    // 4. Noise filtering (NEW)
    let filtered = auto_context::noise::filter(seeds, index);

    // 5. Test file mapping
    let with_tests = if opts.include_tests { enrich_with_tests(&filtered, &index.test_map) } else { filtered };

    // 6. Schema context
    let schema_context = schema::extract_relevant_schema(index.schema.as_ref(), &with_tests);

    // 7. Blast radius
    let blast = if opts.include_blast_radius {
        Some(intelligence::blast_radius::compute_blast_radius(&top_files(&with_tests), &index.graph, &index.pagerank, &index.test_map, 3, opts.focus.as_deref()))
    } else { None };

    // 8. API surface
    let api = intelligence::api_surface::extract_for_context(&with_tests, index);

    // 9. Fill-then-overflow budget allocation (NEW)
    let packed = auto_context::briefing::allocate_and_pack(with_tests, schema_context, api, blast, budget, index);

    // 10. Annotations (already applied by allocate_and_pack via context_quality::annotation)
    packed
}
```

### Fill-Then-Overflow Budget Allocation

Pack content in strict priority order. Each priority level gets as much budget as available after higher priorities are satisfied:

| Priority | Section | Detail Level |
|---|---|---|
| 1 | Target files (highest-scoring) | Full → degrade as needed |
| 2 | Test files (mapped to targets) | Full → degrade as needed |
| 3 | Schema context (tables, columns, FKs) | Documented → Signature |
| 4 | API surface (public signatures) | Signature only |
| 5 | Blast radius preview | Compact summary |

Within each priority level, files are ordered by composite relevance score. Progressive degradation (v0.11.0) applies: if a section doesn't fit at full detail, degrade the lowest-scored files in that section first. If the entire section doesn't fit, include as many files as possible, then move to the next priority level.

## Feature 2: Noise Filtering

### Layer 1: Blocklist

Hardcoded path patterns always excluded from `auto_context` results:

```rust
const NOISE_PATTERNS: &[&str] = &[
    // Vendored / third-party
    "vendor/", "node_modules/", "third_party/", "external/",
    // Build output
    "dist/", "build/", "target/", ".next/", "__pycache__/", "out/",
    // Minified
    ".min.js", ".min.css",
    // Generated code
    ".generated.", "_generated.", ".gen.",
    "_pb.go", "_pb2.py", ".pb.cc", ".pb.h",
    // Lock files (match by exact filename, not substring — "deadlock.rs" must NOT match)
    "package-lock.json", "yarn.lock", "Cargo.lock", "pnpm-lock.yaml", "poetry.lock", "Gemfile.lock", "composer.lock",
    // Source maps
    ".map",
];
```

Additionally, detect generated file markers in the first 5 lines of file content:
- `// Code generated` (Go)
- `# AUTO-GENERATED`, `# This file is auto-generated`
- `/* DO NOT EDIT */`, `// DO NOT EDIT`
- `@generated`

### Layer 2: Similarity Dedup

After scoring, compute Jaccard similarity of symbol names between all candidate file pairs. If similarity > 0.80, keep only the higher-PageRank file.

```rust
pub fn jaccard_symbol_similarity(file_a: &IndexedFile, file_b: &IndexedFile) -> f64 {
    let symbols_a: HashSet<&str> = file_a.parse_result.as_ref()
        .map(|pr| pr.symbols.iter().map(|s| s.name.as_str()).collect())
        .unwrap_or_default();
    let symbols_b: HashSet<&str> = file_b.parse_result.as_ref()
        .map(|pr| pr.symbols.iter().map(|s| s.name.as_str()).collect())
        .unwrap_or_default();
    if symbols_a.is_empty() && symbols_b.is_empty() { return 0.0; }
    let intersection = symbols_a.intersection(&symbols_b).count();
    let union = symbols_a.union(&symbols_b).count();
    if union == 0 { 0.0 } else { intersection as f64 / union as f64 }
}
```

Performance: O(N²) pairwise comparison but N is the candidate set (typically 15-30 files after seed selection), not the full index. Trivial cost.

### Layer 3: Relevance Floor

Default floor: 0.15. Any file scoring below this composite relevance is excluded even if it's a dependency.

```rust
pub const DEFAULT_RELEVANCE_FLOOR: f64 = 0.15;
```

### `filtered_out` Response

Every excluded file appears in the `filtered_out` array with a reason string. This gives the LLM transparency — it can override by calling `pack_context` with the specific file if needed.

### Model Management

**Automatic on first use:**

Model initialization happens at **index build time (server startup)**, not mid-request:

1. `cxpak serve --mcp` → `build_index()` → checks `~/.cxpak/models/all-MiniLM-L6-v2/` for cached model
2. Model not found → download `model.safetensors` + `config.json` + `tokenizer.json` from HuggingFace (~30MB) → cache
3. Model found → load into memory via candle `BertModel::load` (SafeTensors format, NOT ONNX)
4. Embed all public symbol signatures → build embedding index
5. Server ready — all subsequent `auto_context` calls use pre-built index

Download is a one-time ~10 second delay at startup. Subsequent startups load from cache (<1 second). No blocking during MCP requests.

**Bring-Your-Own-Key (optional):** Users can provide their own embedding API key for higher-quality code embeddings. Configured via `.cxpak.json` in the repo root:

```json
{
  "embeddings": {
    "provider": "voyageai",
    "model": "voyage-code-3"
  }
}
```

API key via environment variable: `CXPAK_EMBEDDING_API_KEY=sk-...`

**Full configuration via `.cxpak.json`:**

```json
{
  "embeddings": {
    "provider": "voyageai",
    "model": "voyage-code-3",
    "api_key_env": "VOYAGE_API_KEY",
    "base_url": "https://api.voyageai.com/v1",
    "dimensions": 1024,
```

|---|---|---|---|---|
| `model` | `all-MiniLM-L6-v2` | `text-embedding-3-small` | `voyage-code-3` | `embed-english-v3.0` |

| `base_url` | N/A | `https://api.openai.com/v1` | `https://api.voyageai.com/v1` | `https://api.cohere.com/v2` |

| `batch_size` | 64 | 128 | 128 | 96 |

**This enables:**
- **Custom/self-hosted models** — point `base_url` to a local Ollama instance or any OpenAI-compatible API

- **Dimension override** — for models that support variable dimensions (OpenAI `text-embedding-3-small` supports 256-1536)

**Resolution order:**
1. `.cxpak.json` exists with `embeddings.provider` set + API key available via configured env var → use external provider
2. `.cxpak.json` exists but API key missing → warn, fall back to local MiniLM
3. No `.cxpak.json` → use local MiniLM (auto-download)
4. Local model download fails + no API key → fallback to 6 deterministic signals

When using an external provider, cxpak makes HTTP requests to the provider's API. The embedding index is still cached at `.cxpak/embeddings.bin`. Incremental updates re-embed only changed files.

**The signal weight (0.15) stays the same regardless of provider.** Only the vector quality changes — external providers produce better code embeddings than local MiniLM, so the same 0.15 weight delivers more value.

**Graceful fallback:** If any embedding source fails (local download, API error, network issue), log warning to stderr and continue with 6 deterministic signals. No error returned to MCP client.

```rust
#[cfg(feature = "embeddings")]
pub fn load_or_download_model() -> Option<EmbeddingModel> {
    match load_cached_model() {
        Ok(model) => Some(model),
        Err(_) => match download_model() {
            Ok(model) => Some(model),
            Err(e) => {
                eprintln!("cxpak: embedding model unavailable ({e}), using deterministic signals only");
                None
            }
        }
    }
}
```

### Embedding Index

Built at index construction time (when embeddings are available):

1. For each file in the index, concatenate public symbol signatures into one text
2. Embed the text with MiniLM → 384-dimensional vector
3. Store vectors in `.cxpak/embeddings.bin` (flat file: file_path + vector pairs)

**Incremental updates:** When the file watcher detects changes, re-embed only modified files. Unchanged files keep their existing vectors.

**Query time:** Embed the query text, compute cosine similarity against all file vectors, return as the `embedding_similarity` signal.

### Signal Integration

7th signal in `MultiSignalScorer`:

```rust
pub fn embedding_similarity_signal(
    query_embedding: &[f32],
    file_path: &str,
    embedding_index: &EmbeddingIndex,
) -> SignalResult {
    let score = embedding_index.cosine_similarity(query_embedding, file_path)
        .unwrap_or(0.0);
    SignalResult {
        name: "embedding_similarity",
        score,
        detail: format!("embedding={:.4}", score),
    }
}
```

**Weight configuration — two modes:**

When embeddings active (7 signals):
```rust
SignalWeights {
    path_similarity: 0.15,
    symbol_match: 0.27,
    import_proximity: 0.12,
    term_frequency: 0.16,
    recency_boost: 0.00,
    pagerank: 0.15,
    embedding_similarity: 0.15,
}
// Sum: 1.00
```

When embeddings inactive (6 signals — v0.13.0 values):
```rust
SignalWeights {
    path_similarity: 0.18,
    symbol_match: 0.32,
    import_proximity: 0.14,
    term_frequency: 0.19,
    recency_boost: 0.00,
    pagerank: 0.17,
    embedding_similarity: 0.00,
}
// Sum: 1.00
```

**Field is always present on `SignalWeights` (NOT gated behind `#[cfg(feature = "embeddings")]`).** The field exists unconditionally with value `0.00` when embeddings are inactive. This keeps the struct simple — no conditional compilation on the relevance module. Only the `EmbeddingIndex` on `CodebaseIndex` and the `embedding_similarity_signal()` function are gated behind `#[cfg(feature = "embeddings")]`.

Weight set chosen at `MultiSignalScorer` construction time: if `index.embedding_index.is_some()`, use 7-signal weights; otherwise use 6-signal weights (embedding_similarity = 0.0).

### Implementation

**State architecture:** The `ContextSnapshot` lives in a **separate** `Arc<RwLock<Option<ContextSnapshot>>>` alongside `SharedIndex` in the MCP server state. It is NOT inside `CodebaseIndex` or `SharedIndex`. The `handle_tool_call` function gains an additional parameter: `snapshot: &Arc<RwLock<Option<ContextSnapshot>>>`. The `auto_context` handler takes a write lock to store the snapshot. The `context_diff` handler takes a read lock. This avoids needing a write lock on the main index for snapshot management. For the HTTP server, `ContextSnapshot` is added as a field on `AppState`.

**Snapshot mechanism:** When `auto_context` completes, it stores a snapshot:

```rust
pub struct ContextSnapshot {
    pub timestamp: std::time::Instant,
    pub file_hashes: HashMap<String, u64>,  // path → content hash
    pub symbol_set: HashMap<String, Vec<String>>,  // path → symbol names
    pub edge_set: HashSet<(String, String, String)>,  // (from, to, edge_type)
}
```

**Diff computation:** When `context_diff` is called with `since: "last_call"`:

1. Compare current `index.files` content hashes against snapshot → modified/new/deleted files
2. Compare current symbols against snapshot → new/removed symbols
3. Compare current graph edges against snapshot → added/removed edges
4. Generate human-readable recommendation

When `since` is a git ref: use `cxpak diff` infrastructure (already exists) to compute file-level changes, then enrich with symbol and graph deltas.

**No snapshot exists:** Return a recommendation to call `auto_context` first. This includes after MCP server restart — the snapshot is intentionally in-memory only and does not survive restarts. This is acceptable because: (a) MCP sessions in Claude Code are typically tied to one server process, (b) restarting the server also rebuilds the index, so the previous context is stale anyway, (c) calling `auto_context` again is cheap.

**File watcher re-index:** When `process_watcher_changes` updates the index, the snapshot is NOT cleared. The `context_diff` will correctly detect the changes because it compares current index state against the snapshot hashes.

**Memory cost:** The snapshot is lightweight — file hashes (8 bytes each) + symbol name strings + edge tuples. For a 10,000-file repo, ~1MB. Stored in the MCP server's process memory, not persisted to disk.

## Stable API Guarantee

v1.0.0 establishes semver for the MCP API:

**Stable (no breaking changes in 1.x):**
- MCP tool names
- Required parameters on each tool
- Response structure (field names, types)

**Allowed in 1.x minor releases:**
- New optional parameters on existing tools
- New tools added
- New optional fields in responses
- Performance improvements
- New languages added

**Breaking changes (require 2.0):**
- Removing or renaming MCP tools
- Removing or renaming required parameters
- Changing response field types
- Removing response fields

**NOT part of the public API (can change freely):**
- Internal Rust module structure (`src/index/`, `src/schema/`, etc.)
- CLI output format (markdown structure may evolve)
- `.cxpak/` file formats
- HTTP endpoint paths and response formats (HTTP is convenience, MCP is the contract)

## Design Decisions

| Decision | Choice | Rationale |
|---|---|---|
| One universal response format | Not task-classified | Relevance scoring already handles emphasis — no need for task classifier |
| Fill-then-overflow budget | Not fixed proportions or adaptive | Deterministic, maximizes density, doesn't waste budget on empty sections |
| Noise filtering: 3 layers | Blocklist + dedup + floor | Each catches different noise types; layered defense |
| Relevance floor 0.15 | Not lower or higher | Below 0.15, files are statistically more noise than signal (ICSE 2026 research) |
| Similarity dedup threshold 0.80 | Jaccard on symbol names | Catches v1/v2 copies and generated mocks; doesn't over-filter |
| filtered_out in response | Not silent exclusion | LLM transparency — can override with pack_context |
| Embeddings enabled by default | Auto-download on first use | Zero config philosophy; 30MB is acceptable |
| Graceful embedding fallback | 6 signals when unavailable | Never fail due to embeddings; deterministic core always works |
| MiniLM-L6-v2 model | Not larger model | 30MB, 384-dim, fast inference, good enough for code similarity |
| Embed public signatures only | Not full file content | Signatures are the semantic identity; bodies add noise (ICSE 2026) |
| Weight split 0.15 for embeddings | Approximate reduction of existing 6 (rounded, sums to 1.0) | Meaningful signal without dominating deterministic signals. Exact values prescribed in spec — do not recompute. |
| Context diff via snapshots | Not git-based by default | Faster (hash comparison), works for non-git changes, git fallback available |
| Snapshot in memory only | Not persisted to disk | MCP sessions are typically short-lived; ~1MB is trivial. Cleared on server restart — acceptable since index is also rebuilt. |
| Snapshot in separate Arc<RwLock> | Not inside SharedIndex | Avoids write-locking the main index for snapshot management. auto_context writes snapshot, context_diff reads it. |
| auto_context as tool #10 | Not replacing existing tools | No breaking changes; positioned as primary but not exclusive |
| Stable API: MCP only | CLI output not frozen | MCP is the contract; CLI is convenience |
| v1.0 semver | MCP schema frozen in 1.x | Plugins can depend on stable tool schemas |
