# cxpak v0.11.0 Design: Context Quality

## Summary

Make the context cxpak packs the best possible context for an LLM. Five features in a new `context_quality` module: progressive degradation, concept taxonomy priority, chunk size enforcement, query expansion, and context annotations.

## Architecture

New module: `src/context_quality/`
- `mod.rs` — public API
- `degradation.rs` — detail level rendering, concept priority, budget degradation logic
- `expansion.rs` — query synonym expansion with hierarchical domain detection
- `annotation.rs` — language-aware comment header generation

The budget module (`src/budget/`) calls into `context_quality` for rendering decisions. The index and parser remain unchanged. The output module consumes the degraded+annotated content.

## Feature 1: Progressive Degradation

### Detail Levels

| Level | Name | Content | Approx tokens |
|---|---|---|---|
| 0 | Full | Complete symbol body | Varies |
| 1 | Trimmed | Signature + doc comment + first 20 lines of body + `// ... N more lines` | ~800 |
| 2 | Documented | Signature + doc comment only | ~200 |
| 3 | Signature | Signature only | ~30 |
| 4 | Stub | `fn name(...) // +45 lines` | ~10 |

### Core Types

```rust
// src/context_quality/degradation.rs

#[derive(Debug, Clone, Copy, PartialEq, Eq, PartialOrd, Ord)]
pub enum DetailLevel {
    Full = 0,
    Trimmed = 1,
    Documented = 2,
    Signature = 3,
    Stub = 4,
}

#[derive(Debug, Clone)]
pub struct DegradedSymbol {
    pub symbol: Symbol,
    pub level: DetailLevel,
    pub rendered: String,
    pub rendered_tokens: usize,
    /// Chunk metadata — None if symbol was not split
    pub chunk_index: Option<usize>,  // 0-based
    pub chunk_total: Option<usize>,
    pub parent_name: Option<String>, // original symbol name before splitting
}

#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum FileRole {
    Selected,
    Dependency,
}
```

### Rendering

`render_symbol_at_level(symbol: &Symbol, level: DetailLevel) -> DegradedSymbol`

- Extracts doc comment from body (language-aware):
  - Rust: `///` lines before body
  - Python: `"""..."""` as first statement in body
  - Java/JS/TS/Go/C/C++: `/**..*/` before signature
  - Ruby/Bash/Perl/R: `#` lines before body
  - Haskell: `-- |` or `{- ... -}` before body
  - Elixir: `@doc` attribute
- For Level 1: counts actual body lines, includes first 20, appends `// ... N more lines` omission marker
- For Level 4: generates one-line stub from signature with line count annotation

### Budget Allocation Algorithm

1. Score all files, sort by relevance
2. Assign roles: `Selected` (user-requested via `pack_context`) or `Dependency` (fan-out)
3. **Fast path:** Sum `IndexedFile.token_count` for all target files. If sum fits budget, skip degradation entirely — render all at Level 0. This avoids expensive rendering for small repos or large budgets.
4. **Slow path:** Sum exceeds budget. Render all symbols at Level 0. Compute total tokens.
5. If total fits budget → done (fast path was pessimistic due to token count estimates)
6. Over budget → compute degradation priority for each file:
   ```
   degradation_priority = relevance_score * 0.7 + concept_priority * 0.3
   ```
   Lowest priority degrades first.
7. Degrade `Dependency` files first (lowest priority first), stepping through levels 0→1→2→3→4
8. Then degrade `Selected` files (lowest priority first), but never below Level 2 (Documented)
9. After each degradation step, recompute total tokens. Stop as soon as budget fits.
10. If still over budget after all files at minimum levels → drop lowest-priority `Dependency` files entirely until budget fits.

### Role Constraints

| Role | Minimum Level | Can be dropped? |
|---|---|---|
| Selected | Level 2 (Documented) | No |
| Dependency | Level 4 (Stub) | Yes (last resort) |

## Feature 2: Concept Taxonomy Priority

### Priority Table

| Priority | Value | Concept | SymbolKinds |
|---|---|---|---|
| 1 | 1.00 | Definitions | Function, Method |
| 2 | 0.86 | Structures | Struct, Class, Enum, Interface, Trait, Type, TypeAlias |
| 3 | 0.71 | API Surface | Message, Service, Query, Mutation, Table |
| 4 | 0.57 | Configuration | Key, Block, Variable, Target, Rule, Instruction, Selector, Mixin |
| 5 | 0.43 | Documentation | Heading, Section, Element |
| 6 | 0.29 | Constants | Constant |
| 7 | 0.14 | Imports | (Import entries from ParseResult) |

### Implementation

```rust
pub fn concept_priority(kind: &SymbolKind) -> f64 {
    match kind {
        SymbolKind::Function | SymbolKind::Method => 1.00,
        SymbolKind::Struct | SymbolKind::Class | SymbolKind::Enum
        | SymbolKind::Interface | SymbolKind::Trait
        | SymbolKind::Type | SymbolKind::TypeAlias => 0.86,
        SymbolKind::Message | SymbolKind::Service
        | SymbolKind::Query | SymbolKind::Mutation
        | SymbolKind::Table => 0.71,
        SymbolKind::Key | SymbolKind::Block | SymbolKind::Variable
        | SymbolKind::Target | SymbolKind::Rule
        | SymbolKind::Instruction | SymbolKind::Selector
        | SymbolKind::Mixin => 0.57,
        SymbolKind::Heading | SymbolKind::Section
        | SymbolKind::Element => 0.43,
        SymbolKind::Constant => 0.29,
        _ => 0.14,
    }
}
```

For files with multiple symbols, the file's concept priority is the **maximum** across its symbols. A file with one function and ten imports has priority 1.00 (the function dominates).

### Integration with Degradation

The composite score used for degradation ordering:

```
degradation_priority = relevance_score * 0.7 + concept_priority * 0.3
```

Higher score = survives longer at higher detail. Lower score = degrades first.

## Feature 3: Chunk Size Enforcement

### Constants

```rust
pub const MAX_SYMBOL_TOKENS: usize = 4000;
```

### Splitting Strategy

When a single symbol exceeds `MAX_SYMBOL_TOKENS` at Level 0:

1. **AST-aware split** — re-parse the symbol's source with tree-sitter. Walk the AST to find top-level children of the body node (if/match/loop/block children). Split between them. Each chunk gets the parent signature as a header: `fn handler() { // chunk 2/3`
2. **Blank-line split** — if AST doesn't produce chunks under the limit (e.g., one giant `match` with 200 arms), split at blank lines within the body
3. **Hard split** — last resort, split at the 4000-token boundary on a line break

### Chunk Output

Each chunk becomes its own `Symbol` with:
- Same `kind` and `visibility` as parent
- Name suffixed: `handler [1/3]`, `handler [2/3]`, `handler [3/3]`
- `start_line` / `end_line` adjusted to the chunk's line range
- Parent signature duplicated in each chunk's body (so the LLM knows the containing function)

### When It Runs

During budget allocation, before degradation. The flow:

1. Collect all symbols from target files
2. Split any symbol exceeding `MAX_SYMBOL_TOKENS`
3. Each resulting chunk participates in degradation individually
4. Chunk 1/3 might stay at Level 0 while chunk 3/3 degrades to Level 3

### AST Re-parsing

Oversized symbols are re-parsed on demand. This requires the source text (available from `IndexedFile.content`) but not the original AST tree. tree-sitter re-parsing is fast (~1ms per file) and only triggered for genuinely large symbols.

### Domain Detection

```rust
pub enum Domain {
    Web, Database, Auth, Infra, Testing, Api, Mobile, ML,
}

/// Accepts `&[IndexedFile]` (not `&CodebaseIndex`) so it can be called
/// during index construction before `Self` is fully built.
pub fn detect_domains(files: &[IndexedFile]) -> HashSet<Domain> {
    let mut domains = HashSet::new();

    // Collect file extensions (from paths, not language names — more reliable)
    let extensions: HashSet<String> = files.iter()
        .filter_map(|f| {
            std::path::Path::new(&f.relative_path)
                .extension()
                .and_then(|e| e.to_str())
                .map(|e| e.to_lowercase())
        })
        .collect();

    let paths: Vec<&str> = files.iter()
        .map(|f| f.relative_path.as_str())
        .collect();

    let filenames: HashSet<String> = files.iter()
        .filter_map(|f| {
            std::path::Path::new(&f.relative_path)
                .file_name()
                .and_then(|n| n.to_str())
                .map(|n| n.to_string())
        })
        .collect();

    // Web: .html, .css, .scss, .svelte, .jsx, .tsx present
    if extensions.iter().any(|e| ["html", "css", "scss", "svelte", "jsx", "tsx"].contains(&e.as_str())) {
        domains.insert(Domain::Web);
    }

    // Database: .sql files, migrations/ dir, .prisma files
    if extensions.contains("sql")
        || extensions.contains("prisma")
        || paths.iter().any(|p| p.to_lowercase().contains("migration"))
    {
        domains.insert(Domain::Database);
    }

    // Auth: files matching *auth*, *login*, *session*
    if paths.iter().any(|p| {
        let lower = p.to_lowercase();
        lower.contains("auth") || lower.contains("login") || lower.contains("session")
    }) {
        domains.insert(Domain::Auth);
    }

    // Infra: .tf, .hcl, .tfvars, Dockerfile
    if extensions.iter().any(|e| ["tf", "hcl", "tfvars"].contains(&e.as_str()))
        || filenames.iter().any(|n| n.starts_with("Dockerfile"))
    {
        domains.insert(Domain::Infra);
    }

    // Testing: *test*, *spec* directories
    if paths.iter().any(|p| {
        let lower = p.to_lowercase();
        lower.contains("test") || lower.contains("spec")
    }) {
        domains.insert(Domain::Testing);
    }

    // API: *routes*, *endpoints*, *handler*, openapi*
    if paths.iter().any(|p| {
        let lower = p.to_lowercase();
        lower.contains("route") || lower.contains("endpoint")
        || lower.contains("handler") || lower.contains("openapi")
    }) {
        domains.insert(Domain::Api);
    }

    // Mobile: .swift, .kt, .dart, .xcodeproj, android/
    if extensions.iter().any(|e| ["swift", "kt", "dart"].contains(&e.as_str()))
        || paths.iter().any(|p| p.contains(".xcodeproj") || p.starts_with("android/"))
    {
        domains.insert(Domain::Mobile);
    }

    // ML: .ipynb present AND (*model* OR *train* paths)
    // Requires .ipynb to reduce false positives (web apps have model/ dirs too)
    let has_notebooks = extensions.contains("ipynb");
    let has_ml_paths = paths.iter().any(|p| {
        let lower = p.to_lowercase();
        lower.contains("model") || lower.contains("train")
    });
    if has_notebooks || (has_ml_paths && has_notebooks) {
        domains.insert(Domain::ML);
    }

    domains
}
```

Note: Domain detection uses file extensions from paths (`.tf`, `.sql`, `.html`) and filenames (`Dockerfile`), not the `language` field. Accepts `&[IndexedFile]` so it can be called during `CodebaseIndex::build()` before the struct is fully constructed.

Domain detection runs once at index build time. Result is cached on `CodebaseIndex`.

### Integration with Scoring

The expanded query is: `original_tokens ∪ matching_core_synonyms ∪ matching_active_domain_synonyms`.

**Synonym tokenization:** Each synonym string is added as-is (lowercased) to the expanded set. Synonyms are intentionally single tokens that match `split_identifier()` output — they are chosen to be atomic words that would appear as individual tokens in source code. Compound synonyms like `"memcached"` match as a single token. Multi-word concepts are listed as separate entries (e.g., `"circuit"` and `"breaker"` are separate synonyms, not `"circuit_breaker"`).

Applied to these signals in `MultiSignalScorer`:
- `term_frequency` — expanded tokens used for matching
- `symbol_match` — expanded tokens used for Jaccard similarity

NOT applied to:
- `path_similarity` — exact path segment matching only
- `import_proximity` — structural, not query-dependent
- `recency_boost` — not query-dependent

### Public API

```rust
pub fn expand_query(query: &str, domains: &HashSet<Domain>) -> HashSet<String> {
    let tokens = tokenize(query);
    let mut expanded = tokens.clone();
    for token in &tokens {
        if let Some(synonyms) = CORE_SYNONYMS.get(token.as_str()) {
            expanded.extend(synonyms.iter().map(|s| s.to_string()));
        }
        for domain in domains {
            if let Some(domain_map) = DOMAIN_SYNONYMS.get(domain) {
                if let Some(synonyms) = domain_map.get(token.as_str()) {
                    expanded.extend(synonyms.iter().map(|s| s.to_string()));
                }
            }
        }
    }
    expanded
}
```

## Feature 5: Context Annotations

### Language-Aware Comment Syntax

```rust
pub fn comment_syntax(language: &str) -> (&'static str, &'static str) {
    // Returns (line_prefix, line_suffix)
    // line_suffix is empty for line-comment languages, closing tag for block-comment languages
    match language {
        "rust" | "javascript" | "typescript" | "java" | "go" | "c" | "cpp"
        | "csharp" | "swift" | "kotlin" | "scala" | "dart" | "zig" | "groovy"
        | "objc" | "proto" | "graphql" => ("// ", ""),
        "python" | "ruby" | "bash" | "perl" | "r" | "julia" | "elixir"
        | "yaml" | "toml" | "makefile" | "dockerfile" => ("# ", ""),
        "haskell" | "lua" | "sql" | "ocaml" | "ocaml_interface" => ("-- ", ""),
        "html" | "xml" | "svelte" | "markdown" => ("<!-- ", " -->"),
        "css" | "scss" => ("/* ", " */"),
        "matlab" => ("% ", ""),
        _ => ("// ", ""),
    }
}
```

### Annotation Format

For line-comment languages (e.g., Rust):
```
// [cxpak] src/auth/middleware.rs
// score: 0.72 | role: dependency | parent: src/api/routes.rs
// signals: symbol_match=0.85, path_similarity=0.60, term_frequency=0.71
// detail_level: full (4,200 tokens)
```

For block-comment languages (e.g., HTML):
```
<!-- [cxpak] templates/login.html -->
<!-- score: 0.45 | role: selected -->
<!-- signals: symbol_match=0.30, path_similarity=0.55, term_frequency=0.40 -->
<!-- detail_level: signature (30 tokens) -->
```

Block-comment languages use the same 4-line format as line-comment languages. Each line is wrapped in its own comment block. Signal line (line 3) is still omitted at Level 2+.

### Content Rules

- **Line 1:** `[cxpak]` marker + file path (always present)
- **Line 2:** score (4 decimal places) + role (selected/dependency) + parent file if dependency (always present)
- **Line 3:** signal breakdown with signal names and scores (only at Level 0 and Level 1 — omitted at Level 2+ to save tokens)
- **Line 4:** detail level name + token count (always present — tells the LLM how much detail it's seeing)

### Public API

```rust
pub struct AnnotationContext {
    pub path: String,
    pub language: String,
    pub score: f64,
    pub role: FileRole,
    pub parent: Option<String>,
    pub signals: Vec<SignalResult>,
    pub detail_level: DetailLevel,
    pub tokens: usize,
}

pub fn annotate_file(ctx: &AnnotationContext) -> String
```

### Integration

`pack_context` prepends the annotation before each file's content in the output. The annotation's token count is included in the file's budget allocation — computed via `TokenCounter` and subtracted from available budget before rendering the file's symbols.

## Changes to Existing Modules

### `src/budget/`

The budget allocator gains a new `allocate_with_degradation()` function that uses the `context_quality` module. The existing `allocate()` continues to work for `overview` and `diff` commands. The new function is used by `pack_context` and `context_for_task`.

### `src/relevance/signals.rs`

- Make `tokenize()` public: `pub fn tokenize(s: &str) -> HashSet<String>` — needed by `context_quality/expansion.rs` and ensures consistent tokenization across modules.
- `term_frequency()` and `symbol_match()` accept an optional `expanded_tokens: Option<&HashSet<String>>` parameter. When provided, the expanded set is used instead of calling `tokenize(query)`. Existing call sites pass `None` (unchanged behavior).

### `src/relevance/mod.rs`

`MultiSignalScorer` gains:
- An optional `expanded_tokens: Option<HashSet<String>>` field, set via `with_expansion(tokens: HashSet<String>)` builder method.
- When set, `score()` passes `Some(&self.expanded_tokens)` to `term_frequency` and `symbol_match`.
- `score_all()` automatically uses stored expanded tokens if present — no separate call needed.

### `src/index/mod.rs`

`CodebaseIndex` gains a `domains: HashSet<Domain>` field populated at build time by `detect_domains()`. Cached for the lifetime of the index.

### `src/commands/serve.rs`

`cxpak_pack_context` handler updated to:
1. Use `allocate_with_degradation()` instead of simple token counting
2. Prepend annotations to each file's content
3. Report `detail_level` per file in the response JSON

`cxpak_context_for_task` handler updated to:
1. Run query expansion before scoring
2. Pass expanded tokens to `MultiSignalScorer`

## Design Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Hybrid degradation model | Index stores whole symbols, budget degrades, oversized symbols split | Full data always available for trace/search; splitting is last resort |
| 5 detail levels | Full → Trimmed → Documented → Signature → Stub | Granular control; each level is meaningfully different |
| Role-based constraints | Selected ≥ Level 2, Dependency can drop | Respects user intent; dependencies are supporting context |
| Composite degradation score | 0.7 relevance + 0.3 concept | Relevance dominates but concept type breaks ties |
| File concept priority = max symbol | Highest symbol kind wins | A file with one function and ten imports is a "function file" |
| AST re-parse on demand | Only for oversized symbols | Avoids storing ASTs in index; tree-sitter is fast (~1ms) |
| Max 4000 tokens per symbol | Matches typical LLM attention window for a single unit | Larger units lose coherence; smaller wastes splitting effort |
| Core + domain synonym structure | Always-on core, heuristic-activated domains | Zero config; adapts to repo automatically |
| Domain detection at index build | Cached on CodebaseIndex | Runs once, not per query |
| Expansion on TF + symbol, not path | Path matching should be exact | "auth" should match symbol `authenticate` but not path `authorization/config.toml` by synonym |
| Language-aware annotations | Correct comment syntax per language | LLM sees valid syntax; no parse errors in packed context |
| Signal line omitted at Level 2+ | Save tokens at high degradation | When space is tight, signals are less useful than content |
| Annotation tokens in budget | Count toward file allocation | Honest accounting; prevents budget overrun |
| `.pl` maps to Perl | Known ambiguity (Prolog also uses `.pl`) | Perl is far more common; same pattern as `.m` → Obj-C |
