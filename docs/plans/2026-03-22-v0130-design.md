# cxpak v0.13.0 Design: Intelligence

## Summary

Add structural intelligence to cxpak: PageRank-based symbol importance scoring, blast radius analysis for change impact, API surface extraction, and test file mapping. All four features are deterministic, computed from the existing typed dependency graph, and require zero configuration.

## Architecture

New module: `src/intelligence/`
- `mod.rs` — public API, re-exports
- `pagerank.rs` — file-level PageRank computation, within-file symbol importance
- `blast_radius.rs` — change impact analysis via reverse BFS on typed graph
- `api_surface.rs` — public symbol extraction + HTTP route detection (12 frameworks)
- `test_map.rs` — test file ↔ source file mapping via naming conventions + import analysis

The intelligence module reads from `CodebaseIndex` (including `DependencyGraph` with typed edges from v0.12.0 and `SchemaIndex`) and produces derived data structures cached on the index. Two new MCP tools (`cxpak_blast_radius`, `cxpak_api_surface`) are added. PageRank and test mapping enhance existing tools internally.

## Feature 1: PageRank Symbol Importance

### File-Level PageRank

Standard PageRank algorithm on the typed `DependencyGraph`:

```rust
pub fn compute_pagerank(
    graph: &DependencyGraph,
    damping: f64,
    iterations: usize,
) -> HashMap<String, f64>
```

- Damping factor: 0.85 (standard)
- Max iterations: 100 (converges well before this for typical repos)
- Convergence threshold: 1e-6 (stop early if delta < threshold)
- All edge types contribute equally (an Import edge and a ForeignKey edge both transfer rank)
- Scores normalized to 0.0–1.0 range
- Traversal direction: **forward edges only** (standard PageRank — if A imports B, A transfers rank to B, so heavily-imported files rank highest)
- Computed once at index build time, cached on `CodebaseIndex` as `pub pagerank: HashMap<String, f64>`

**Build-time graph requirement:** `DependencyGraph` is currently built on-demand at each call site, NOT stored on `CodebaseIndex`. For v0.13.0, the graph must be built during `CodebaseIndex::build()` / `build_with_content()` and stored as a new field:

```rust
pub struct CodebaseIndex {
    // ... existing fields ...
    pub pagerank: HashMap<String, f64>,
    pub test_map: HashMap<String, Vec<TestFileRef>>,
    pub graph: DependencyGraph,  // NEW — cached for PageRank, test mapping, and reuse by callers
}
```

**Build-time ordering:**
1. Build base index (files, parse results, term frequencies, domains)
2. Build schema index (`Option<SchemaIndex>`)
3. Build dependency graph (with schema edges)
4. Compute PageRank from graph
5. Compute test mapping from graph + naming conventions
6. Store all on `CodebaseIndex`

Existing call sites that build the graph on-demand (`trace.rs`, `diff.rs`, `serve.rs`, `overview.rs`, `seed.rs`) should be updated to use `index.graph` instead. This eliminates redundant graph construction.

### Within-File Symbol Importance

No new graph needed. For each symbol in a file:

```
symbol_importance = file_pagerank * symbol_weight
```

Where `symbol_weight`:
- **1.0** — public symbol whose name appears in at least one other file's `term_frequencies`
- **0.7** — public symbol not referenced in other files
- **0.3** — private symbol

This uses `term_frequencies` (from v0.9.0) which maps each file to a bag-of-words. If symbol name "authenticate" appears in `term_frequencies["src/api/routes.rs"]`, the symbol is cross-referenced.

**Cross-reference check algorithm:** At index build time, build an inverted index: `symbol_name → set of files containing it`. This is O(N*M) to build once (N files, M avg terms per file) but O(1) per lookup. For a 10,000-file repo with ~100 terms per file, this is 1M entries — trivial memory. The inverted index is a `HashMap<String, HashSet<String>>` built by iterating all term_frequencies once.

```rust
pub fn build_symbol_cross_refs(
    term_frequencies: &HashMap<String, HashMap<String, u32>>,
) -> HashMap<String, HashSet<String>>  // symbol_name → files containing it

pub fn symbol_importance(
    symbol: &Symbol,
    file_pagerank: f64,
    cross_refs: &HashMap<String, HashSet<String>>,
    file_path: &str,
) -> f64 {
    let is_public = symbol.visibility == Visibility::Public;
    let is_referenced = cross_refs.get(&symbol.name.to_lowercase())
        .map(|files| files.iter().any(|f| f != file_path))
        .unwrap_or(false);
    let weight = match (is_public, is_referenced) {
        (true, true) => 1.0,
        (true, false) => 0.7,
        (false, _) => 0.3,
    };
    file_pagerank * weight
}
```

### Integration as Signal #6

New signal in `MultiSignalScorer`:

```rust
pub struct SignalWeights {
    pub path_similarity: f64,    // 0.18 (was 0.22)
    pub symbol_match: f64,       // 0.32 (was 0.38)
    pub import_proximity: f64,   // 0.14 (was 0.17)
    pub term_frequency: f64,     // 0.19 (was 0.23)
    pub recency_boost: f64,      // 0.00 (unchanged)
    pub pagerank: f64,           // 0.17 (new)
}
```

Weights redistributed proportionally to sum to 1.0. PageRank gets 0.17 — meaningful but not dominant. Existing signals still drive most of the score; PageRank breaks ties in favor of structurally important code.

The `pagerank` signal implementation:

```rust
pub fn pagerank_signal(file_path: &str, pagerank: &HashMap<String, f64>) -> SignalResult {
    let score = pagerank.get(file_path).copied().unwrap_or(0.0);
    SignalResult {
        name: "pagerank",
        score,
        detail: format!("pagerank={:.4}", score),
    }
}
```

### Integration with Budget Degradation (v0.11.0)

The `allocate_with_degradation()` composite score becomes:

```
degradation_priority = relevance_score * 0.6 + concept_priority * 0.2 + pagerank * 0.2
```

Updated from `relevance * 0.7 + concept * 0.3` to give PageRank a 20% say in what survives degradation. High-PageRank symbols (heavily referenced infrastructure) survive longest.

**Threading PageRank into the function:** `allocate_with_degradation()` gains an optional `pagerank: Option<&HashMap<String, f64>>` parameter. When present, the composite score uses the 0.6/0.2/0.2 formula. When `None` (backwards compatibility), falls back to the v0.11.0 formula (0.7/0.3). Callers pass `index.pagerank` (or `Some(&index.pagerank)`).

## Feature 2: Blast Radius MCP Tool

### Schema

```json
{
  "name": "cxpak_blast_radius",
  "description": "Analyze the impact of changing specified files. Returns affected files categorized by impact type with risk scores.",
  "inputSchema": {
    "type": "object",
    "properties": {
      "files": {
        "type": "array",
        "items": { "type": "string" },
        "description": "File paths that are changing"
      },
      "depth": {
        "type": "number",
        "description": "Max dependency hops to follow (default 3)",
        "default": 3
      },
      "focus": {
        "type": "string",
        "description": "Path prefix to scope results"
      }
    },
    "required": ["files"]
  }
}
```

### Response Format

```json
{
  "changed_files": ["src/auth/middleware.rs"],
  "total_affected": 17,
  "categories": {
    "direct_dependents": [
      { "path": "src/api/routes.rs", "edge_type": "import", "risk": 0.92 },
      { "path": "src/api/admin.rs", "edge_type": "import", "risk": 0.85 }
    ],
    "transitive_dependents": [
      { "path": "src/main.rs", "edge_type": "import", "hops": 2, "risk": 0.61 }
    ],
    "test_files": [
      { "path": "tests/test_auth.rs", "risk": 0.95, "note": "directly tests changed file" }
    ],
    "schema_dependents": [
      { "path": "schema/tables.sql", "edge_type": "embedded_sql", "risk": 0.70 }
    ]
  },
  "risk_summary": {
    "high": 4,
    "medium": 8,
    "low": 5
  }
}
```

### Categorization Logic

For each affected file, assign to ONE category (highest priority wins):

1. **test_files** — file path matches test patterns (contains `test`, `spec`, `__tests__`) AND is mapped to a changed file via test mapping
2. **direct_dependents** — 1-hop reverse edge from a changed file
3. **schema_dependents** — connected via ForeignKey, EmbeddedSql, OrmModel, ViewReference, TriggerTarget, or IndexTarget edge types
4. **transitive_dependents** — 2+ hops away via any edge type

### Risk Scoring

```rust
pub fn compute_risk(
    hops: usize,
    edge_type: &EdgeType,
    file_pagerank: f64,
    has_test_coverage: bool,
) -> f64 {
    let hop_decay = 1.0 / (hops as f64 + 1.0);
    let edge_weight = match edge_type {
        EdgeType::Import | EdgeType::ForeignKey | EdgeType::OrmModel => 1.0,
        EdgeType::EmbeddedSql | EdgeType::ViewReference | EdgeType::FunctionReference => 0.8,
        EdgeType::TriggerTarget | EdgeType::IndexTarget => 0.6,
        EdgeType::MigrationSequence => 0.5,
    };
    let test_penalty = if has_test_coverage { 1.0 } else { 1.2 };
    let raw = hop_decay * edge_weight * file_pagerank * test_penalty;
    raw.clamp(0.0, 1.0)
}
```

Risk thresholds: high ≥ 0.7, medium ≥ 0.3, low < 0.3.

When multiple edges connect a changed file to an affected file (e.g., Import + EmbeddedSql), the highest-risk edge wins.

### Implementation

BFS on `DependencyGraph` starting from changed files, following reverse edges (`dependents()`). Track hops count per file. At each visited file:
1. Compute risk score
2. Categorize
3. Check test mapping for test file classification
4. Respect `depth` limit and `focus` prefix

Also exposed as `POST /blast_radius` on the HTTP server.

```json

  "symbols": {

  "graphql_types": [],

```

### Symbol Extraction

Filter to `Visibility::Public` symbols only. For each, include:
- `name`, `kind`, `signature`
- Doc comment (extracted same as v0.11.0 `render_documented`)
- No bodies — signatures only

Files sorted by PageRank descending. Within each file, symbols sorted by `concept_priority` (v0.11.0) — definitions first.

### Route Detection (12 Frameworks)

Each framework is a regex pattern applied to symbol bodies or file content:

| Framework | Language | Regex Pattern |
|---|---|---|
| Express/Koa | JS/TS | `(app\|router)\.(get\|post\|put\|delete\|patch)\s*\(\s*["']/` |
| Flask | Python | `@(app\|blueprint)\.(route\|get\|post\|put\|delete)\s*\(\s*["']/` |
| Django | Python | `path\s*\(\s*["']` in files matching `*urls*.py` |
| FastAPI | Python | `@(app\|router)\.(get\|post\|put\|delete\|patch)\s*\(\s*["']/` |
| Spring | Java/Kotlin | `@(Get\|Post\|Put\|Delete\|Patch\|Request)Mapping\s*\(\s*["']/` |
| actix-web | Rust | `#\[(get\|post\|put\|delete\|patch)\s*\(\s*["']/` |
| axum | Rust | `\.route\s*\(\s*["']/` |
| Gin | Go | `(r\|router\|group)\.(GET\|POST\|PUT\|DELETE\|PATCH)\s*\(\s*["']/` |
| Echo | Go | `(e\|g\|echo\|group)\.(GET\|POST\|PUT\|DELETE\|PATCH)\s*\(\s*["']/` |
| Rails | Ruby | `(get\|post\|put\|patch\|delete)\s+["']/` in files matching `*routes*` |
| ASP.NET | C# | `\[(Http(Get\|Post\|Put\|Delete\|Patch)\|Route)\s*\(\s*["']/` |
| Phoenix | Elixir | `(get\|post\|put\|patch\|delete)\s+["']/` in files matching `*router*` |

Each regex extracts: HTTP method, URL path, enclosing function/symbol name as handler, file path, line number.

Framework auto-detection: try all 12 patterns, report whichever framework(s) produce matches. No configuration needed.

### gRPC and GraphQL

Already extracted by v0.10.0 parsers:
- Proto files: `SymbolKind::Service`, `SymbolKind::Method` → include as gRPC services
- GraphQL files: `SymbolKind::Query`, `SymbolKind::Mutation`, `SymbolKind::Type` → include as GraphQL types

### Token Budget

Apply v0.11.0 degradation if output exceeds budget. Highest-PageRank files survive; lowest get dropped. Within files, `concept_priority` determines which symbols survive.

Also exposed as `POST /api_surface` on the HTTP server.

## Feature 4: Test File Mapping

### Naming Convention Matching

For each source file in the index, check if test file variants exist:

| Source Pattern | Test Patterns Checked |
|---|---|
| `src/foo.rs` | `tests/foo.rs`, `tests/foo_test.rs`, `src/foo_test.rs`, `tests/test_foo.rs` |
| `src/foo.py` | `tests/test_foo.py`, `tests/foo_test.py`, `test_foo.py` |
| `src/Foo.java` | `src/FooTest.java`, `test/FooTest.java`, `tests/FooTest.java` |
| `src/foo.ts` | `src/foo.test.ts`, `src/foo.spec.ts`, `tests/foo.test.ts`, `__tests__/foo.test.ts` |
| `src/foo.go` | `src/foo_test.go` (same directory, Go convention) |
| `lib/foo.rb` | `spec/foo_spec.rb`, `test/foo_test.rb`, `spec/lib/foo_spec.rb` |

General strategy (catch-all for all 42 languages including C, C++, C#, Swift, Kotlin, etc. not in the table above): strip directory prefix + extension, try common test prefixes (`test_`), suffixes (`_test`, `_spec`, `.test`, `.spec`, `Test`, `Tests`), and test directories (`tests/`, `test/`, `spec/`, `__tests__/`). The explicit table covers major language conventions; the general strategy handles the rest.

### Import Analysis

For each file identified as a test file (path contains `test`, `spec`, `__tests__`), examine its `ParseResult.imports`. Each imported source module creates a mapping.

```rust
pub struct TestMapping {
    pub source_file: String,
    pub test_files: Vec<TestFileRef>,
}

pub struct TestFileRef {
    pub path: String,
    pub confidence: TestConfidence,
}

#[derive(Debug, Clone, PartialEq, Eq, Serialize, Deserialize)]
pub enum TestConfidence {
    NameMatch,
    ImportMatch,
    Both,
}
```

### Storage

Cached on `CodebaseIndex` as:
```rust
pub test_map: HashMap<String, Vec<TestFileRef>>
```

Keyed by source file path, values are test files that cover it. Computed at index build time alongside PageRank.

### Integration with Existing Tools

**`pack_context`:** New optional parameter `include_tests: bool` (default `true`). When true and a packed file has test mappings, include test files automatically with `"included_as": "test_file"`.

**`blast_radius`:** Test files appear in the `test_files` category. Files without test mappings get the `test_coverage_penalty` in risk scoring.

**`context_for_task`:** Files without test coverage get a 5% scoring penalty (multiplier 0.95 on the composite score). This slightly deprioritizes untested code — the LLM should focus on code that has tests to validate against.

## Design Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Hybrid PageRank (file + symbol heuristic) | Not pure symbol-level | Symbol-level call graph doesn't exist and building one is a massive undertaking |
| PageRank as signal #6 | Not replacing import_proximity | PageRank measures transitive importance; import_proximity measures direct connections. Both useful. |
| Weight redistribution | Proportional reduction of existing weights | Preserves relative ranking of existing signals |
| PageRank weight 0.17 | Not higher | Should break ties, not dominate. Structural importance is one factor among many. |
| Degradation formula updated | 0.6/0.2/0.2 split | Give PageRank equal weight with concept priority in degradation decisions |
| Blast radius via reverse BFS | Not forward BFS | "What breaks if I change this?" = reverse graph walk |
| Risk scoring formula | Multiplicative factors | Each factor independently reduces risk; combined effect is intuitive |
| Test coverage penalty 1.2x | Not higher | Slight nudge, not a dramatic overweight. Untested code is riskier but not always. |
| Categorization: highest priority wins | Not multi-category | Each file in exactly one category — cleaner for LLM reasoning |
| API surface: signatures only | No bodies | ICSE 2026 research: API info is highest-value context. Bodies add noise. |
| 12 route frameworks | Not fewer | Each is one regex; incremental cost is minimal |
| Framework auto-detection | Not configured | Try all patterns, report matches. Zero config. |
| Test mapping: naming + imports | Not content analysis | Content matching on function names creates too many false links |
| `include_tests` default true | Not false | Tests are almost always useful context. Opt-out rather than opt-in. |
| Untested file scoring penalty 5% | Not higher | Gentle bias; some intentionally untested code (generated, vendored) should not be punished hard |
| New module `src/intelligence/` | Not in existing modules | Clean separation; these features are a distinct concern from context_quality or schema |
| Route regex patterns in spec use markdown-escaped pipes | Implementer must use raw `\|` → `|` | Markdown table limitation; spec patterns are illustrative, not copy-paste |
| PageRank direction: forward edges | Standard interpretation | If A imports B, rank flows A→B; B (the imported file) ranks highest |
| Graph cached on CodebaseIndex | Eliminates redundant on-demand construction | All callers (trace, diff, serve, overview, seed) use `index.graph` instead of building their own |
| Build-time ordering: index → schema → graph → pagerank → test_map | Sequential dependencies | Each step depends on the previous; all complete before index is returned to callers |
