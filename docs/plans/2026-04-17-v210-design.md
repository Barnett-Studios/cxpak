# cxpak v2.1.0 "The Polish" — Design Spec

> **Thesis:** v2.0.0 shipped the intelligence engine and 6 views as separate HTML files. v2.1.0 makes them feel like a single premium tool: one page, one shared state, one command palette, working stubs across the entire platform. No new intelligence — every signal already exists. This release is pure experience and platform completeness.

**Current state:** v2.0.0 shipped. 2,436 tests, 0 failures, clippy clean. ~9,700 lines of new Rust in v2.0.0 across `src/visual/` and `src/plugin/`. 6 standalone HTML view files. 9/12 v1 API endpoints and 11/14 LSP methods still return stub responses.

**The three "known v2.0.0 bugs" flagged in the v2.0.0 post-release validation (onboarding `.take(3)`, missing test-file exclusion, MCP >1MB inline) have already been fixed on main** — verified by grep against the current working tree as of 2026-04-17. Pillar 3 from the brainstorm dissolves; this release focuses on Pillars 1 and 2 only.

**Goal:** Transform cxpak from "impressive technical demo" into "tool I open every day." Ship the single-page dashboard and wire every platform stub to real intelligence.

## Pillar 1: Single-Page Dashboard

**"SPA" (Single-Page Application) in this spec** means a browser-architecture SPA: one HTML file containing all six views (Dashboard, Architecture Explorer, Risk Heatmap, Flow Diagram, Time Machine, Diff View), with client-side hash routing that shows/hides view containers. The user can move between views without a page reload. This replaces v2.0.0's approach of emitting six separate HTML files (one per view, each self-contained) with a single unified file.

**Terminology used throughout Pillar 1:**

| Term | Meaning |
|---|---|
| **View** | One of the six user-facing modes: dashboard / architecture / risk / flow / timeline / diff. Each has a dedicated `<section id="view-X">` in the DOM. |
| **Section** | A numbered subsection of this spec document (e.g. § 1.3). Never used for DOM elements. |
| **Panel** | The sliding right-side drawer (Inspector Panel, § 1.5). One panel exists; it is re-populated per node click. |
| **Overlay** | A full-screen modal (Command Palette, Help Overlay). Layered over everything. |
| **Data tag** | A `<script type="application/json">` element embedded in the body of the HTML, containing pre-serialized view data read at bootstrap. |
| **Controller** | The JS code in `assets/cxpak-spa-controller.js` that wires up routing, palette, inspector, theme, etc. |

### 1.1 Architecture

**One HTML file. Hash routing. All view data embedded.** Opens instantly from `file://`. No build step, no npm, no server.

The existing per-view renderers (`render_dashboard`, `render_architecture_explorer`, `render_risk_heatmap`, `render_flow_diagram`, `render_time_machine`, `render_diff_view`) remain unchanged. They continue to emit individual HTML files — useful for embedding one view in a report. v2.1.0 adds a NEW top-level renderer, `render_spa`, that produces the unified dashboard.

**Document structure:**

```
<!DOCTYPE html>
<html lang="en" data-theme="dark">
<head>
  <title>cxpak — {repo_name}</title>
  <style>{INLINED CSS}</style>
</head>
<body>
  <div id="cxpak-app">
    <header id="cxpak-header">
      <!-- logo, repo name, nav tabs, theme toggle, freshness badge -->
    </header>
    <main id="cxpak-main">
      <section id="view-dashboard"    class="cxpak-view"></section>
      <section id="view-architecture" class="cxpak-view" hidden></section>
      <section id="view-risk"         class="cxpak-view" hidden></section>
      <section id="view-flow"         class="cxpak-view" hidden></section>
      <section id="view-timeline"     class="cxpak-view" hidden></section>
      <section id="view-diff"         class="cxpak-view" hidden></section>
    </main>
    <aside id="cxpak-inspector" class="cxpak-inspector" hidden>
      <!-- Context-sensitive detail panel, opens on node click -->
    </aside>
  </div>
  <div id="cxpak-palette-overlay" class="cxpak-palette-overlay" hidden>
    <!-- Command palette modal -->
  </div>

  <!-- Embedded data, one <script> tag per view + shared data -->
  <script id="cxpak-meta"               type="application/json">{...}</script>
  <script id="cxpak-dashboard-data"     type="application/json">{...}</script>
  <script id="cxpak-architecture-data"  type="application/json">{...}</script>
  <script id="cxpak-risk-data"          type="application/json">{...}</script>
  <script id="cxpak-timeline-data"      type="application/json">{...}</script>
  <script id="cxpak-search-index"       type="application/json">[...]</script>

  <!-- D3 bundle inlined -->
  <script>{D3_BUNDLE}</script>
  <!-- SPA controller inlined -->
  <script>{SPA_CONTROLLER_JS}</script>
</body>
</html>
```

Flow and Diff views have empty data by default (they require a `--symbol` or `--files` parameter). The SPA shows an empty-state card in those views with explicit, copyable wording:

- **Flow view empty state:** "Flow diagram requires a symbol. Run: `cxpak visual --visual-type flow --symbol <name>`." The command is in a `<code>` block with a copy-to-clipboard button.
- **Diff view empty state:** "Diff view requires file paths. Run: `cxpak visual --visual-type diff --files <a>,<b>`." Same treatment.

These exact strings are specified to prevent two engineers writing inconsistent messages.

**Data tag emission rule (always-emit invariant).** Every data tag (`cxpak-dashboard-data`, `cxpak-architecture-data`, `cxpak-risk-data`, `cxpak-timeline-data`, `cxpak-flow-data`, `cxpak-diff-data`, `cxpak-meta`, `cxpak-search-index`) is **always present in the DOM**, even when the corresponding data is unavailable. **Empty representation is uniformly `null`** across every view (not `[]` for some and `null` for others):

| Data state | Tag content |
|---|---|
| Data available | Full JSON payload |
| Dashboard / Architecture / Risk — empty index | Full JSON payload with empty inner arrays (e.g. `{"risks": {"top_risks": []}, ...}`) |
| Timeline — no git cache | `null` |
| Flow — no `--symbol` provided | `null` |
| Diff — no `--files` provided | `null` |
| Search index — zero entries | `[]` (search index is a list by type; `null` would change its type semantics) |
| Meta — always | Full payload (never null) |

The controller always calls `JSON.parse(document.getElementById('cxpak-X-data').textContent)` and branches on `null` to render the empty-state card. `document.getElementById(...)` returning `null` (tag missing entirely) is reserved for "implementation bug" and surfaces as a thrown error in bootstrap.

**Injection-safety enforcement across ALL tags.** Contract for the spec: every single `<script type="application/json">` tag emitted by `render_spa()` passes through `escape_script_tag()` without exception. Enforced by a Rust integration test (`tests/spa_all_tags_escaped.rs`): render SPA on a fixture index containing a file path literally `</script><img src=x onerror=alert(1)>.rs`, grep the output HTML for unescaped `</script>` occurrences between `<script id="cxpak-` and `</script>` closing boundaries — must find only the legitimate closings, never an embedded one. This catches a future developer adding a new data tag but forgetting the escape call.

### 1.2 SPA Controller Architecture

The controller JS is a single JavaScript asset file — `assets/cxpak-spa-controller.js` — included via `include_str!`, same pattern as `assets/d3-bundle.min.js` and `assets/cxpak-visual.css`. It is NOT a Rust string literal.

**Rationale:** A 300–500 line string literal embedded in Rust is unlintable, unreadable in code review, produces terrible sourcemaps in browser devtools, and makes it impossible to use editor JS tooling (prettier, eslint). An asset file fixes all four problems. The asset is still inlined into the final HTML at render time, so the "self-contained HTML" contract is preserved.

The controller file is organized into 6 logical sections separated by comment headers, each ~50–150 lines:

1. **Bootstrap** — reads all JSON data scripts, builds `CX.state` global
2. **Router** — `hashchange` listener, toggles `hidden` attribute on view sections, initializes a view's D3 renderer on first visit (lazy)
3. **Command palette** — Cmd+K/Ctrl+K handler, fuzzy search, result navigation
4. **Inspector panel** — slide-in sidebar on node click, populated from node metadata
5. **Theme toggle** — reads/writes `localStorage["cxpak-theme"]`, toggles `data-theme` attribute on `<html>`
6. **Keyboard navigation** — roving tabindex on focusable nodes, arrow/enter/escape

**Visibility changes in `src/visual/render.rs`:** The existing private functions `common_js()`, `view_controller_js()`, and `escape_script_tag()` must be promoted to `pub(crate)` so `src/visual/spa.rs` can reuse them. No functional changes — these functions already exist and work correctly. The visibility change is mechanical.

**Existing data-builder functions are reused unchanged:** `build_dashboard_data()`, `build_architecture_explorer_data()`, `build_risk_heatmap_data()`, `build_flow_diagram_data()`, `build_time_machine_data()`, `build_diff_view_data()` are already pure data functions separate from HTML emission (verified at `src/visual/render.rs:1253`, `1537`, `1717`, etc.). `render_spa()` calls these directly and embeds their JSON output — no refactor needed.

### 1.3 Hash Router

| Hash | View | Init function |
|---|---|---|
| `#dashboard` (default) | Dashboard | `initDashboard()` |
| `#architecture` | Architecture Explorer | `initArchitecture()` |
| `#risk` | Risk Heatmap | `initRisk()` |
| `#flow` | Flow Diagram | `initFlow()` |
| `#timeline` | Time Machine | `initTimeline()` |
| `#diff` | Diff View | `initDiff()` |

Unknown hashes redirect to `#dashboard`. Router runs on `DOMContentLoaded` and on every `hashchange`. Each `init*` function is idempotent — subsequent calls re-use the existing SVG if present, only rebind data if changed.

**D3 transition cleanup on view switch.** Before toggling `hidden` on the departing view, the router calls `d3.selectAll('#view-<id> *').interrupt()` to cancel any in-flight transitions. Without this, animations continue running in hidden DOM, wasting CPU and producing unexpected visual state on return.

**Inspector state on view switch.** Switching views via the router ALWAYS closes the inspector (`CX.state.inspector = null`, element gains `hidden` attribute). Content in the inspector is view-specific — leaving it open on the wrong view confuses users.

**Programmatic hash updates use `history.pushState`, never `location.hash = ...`.** Assigning `location.hash` fires `hashchange` synchronously within the same call stack, causing re-entrant `init*` calls. `history.pushState` updates the URL without firing `hashchange`; the router only responds to `hashchange` for external navigation (browser back/forward, URL-bar edits). Tests covering drill-down must use `window.dispatchEvent(new HashChangeEvent('hashchange'))` to simulate external navigation.

**Architecture drill-down depth is reflected in the hash.** This matters because users expect the browser Back button to work. Hash schema:

- `#architecture` — Level 1 (module graph)
- `#architecture?module=<prefix>` — Level 2 (files within module)
- `#architecture?file=<path>` — Level 3 (symbols within file)
- Other views: single hash only (no depth levels)

The `hashchange` handler inspects query params and dispatches to the correct level initializer. Navigating from L2 → L1 via Back button triggers `hashchange`, which calls `initArchitecture()` at L1.

**Query-string parameters** on the hash are parsed into `CX.state.focus` / `CX.state.module` / `CX.state.file` / `CX.state.symbol` / `CX.state.files`. When a view initializes with a non-empty focus/file/symbol value: (a) pan the graph so the matching node is centered, (b) apply a 2-second `--accent-blue` ring highlight on that node, (c) open the inspector for that node automatically. If no node matches, show a non-blocking toast: `"No node found for <param>"`.

**Router param sanitization (security invariant).** Before any router-derived value is read into `CX.state` or rendered to the DOM, it is validated against `^[A-Za-z0-9._/\-]{1,512}$` in JS. Values failing validation are replaced with an empty string and a toast shows `"Invalid route parameter; ignoring."` This is a defense-in-depth invariant: values reach the DOM only through `textContent` or `setAttribute`, but sanitization at the boundary prevents misuse by future code. A regression test loads `#architecture?focus=<script>alert(1)</script>` and asserts the script string is not present anywhere in the DOM tree.

### 1.4 Command Palette

Triggered by **Cmd+K** (macOS) or **Ctrl+K** (Linux/Windows). Press Escape to close. Arrow keys navigate results, Enter activates.

**Focus capture.** On palette open, `paletteInput.focus()` is called immediately in the same event handler (no setTimeout), and the element that previously had focus is stored in `CX.state.prePaletteFocus`. On palette close, `CX.state.prePaletteFocus?.focus()` restores it. If the previously-focused element has been removed from the DOM, fall back to the first nav tab.

**Searchable entries:**

```rust
pub struct SearchEntry {
    pub label: String,       // e.g. "src/auth/jwt.rs" or "validateToken"
    pub kind: String,        // "file" | "symbol" | "module" | "view"
    pub context: String,     // For symbols: owning file. For files: language. Else "".
    pub detail: String,      // Secondary text: "rust · 420 tokens · PR 0.042"
    pub target: String,      // Hash target: "#architecture?file=src/auth/jwt.rs"
}
```

The search index is computed once in Rust at HTML generation time and embedded as JSON. At runtime, the palette performs **subsequence fuzzy match** (characters must appear in order but not necessarily contiguously) plus a simple ranker: prefix matches rank first, then contiguous-substring matches, then subsequence matches. Max 50 results displayed.

**Scope:** all files, all public symbols, all module prefixes, all 6 views. Deduplicated.

**Definition of "module prefix":** the first two path segments of each file, matching the existing `module_depth: 2` default used by `ArchitectureMap`. E.g. `src/auth/jwt.rs` → module prefix `src/auth`. Top-level files (e.g. `Cargo.toml`, with no second segment) are excluded from the module-prefix sub-list but still appear as `kind: "file"` entries.

**Scale cap.** If `build_search_index()` would produce more than **20,000 entries**, the list is truncated: include all 6 view entries, all module prefixes, then files and symbols ranked by PageRank descending until the cap is reached. Emit a `cxpak warn` log line: `"Search index capped at 20,000 entries (repo has N total). Lowest-PageRank files and symbols excluded."` This prevents O(n) palette scans from stalling the UI thread on 50k-file monorepos.

**Entry ordering (determinism contract):** `build_search_index()` must sort its output by `(kind, label, context)` before returning. The tertiary `context` key is required because two symbols with the same name in different files would otherwise have unspecified order. Sorting happens once in Rust; the runtime palette then re-sorts by match score for display.

**Search ranking.** The palette computes a composite sort key per entry and sorts by descending key. Each key is a `(tier_u8, match_start_i32_neg, kind_rank_u8, label_len_usize, label_str, context_str)` tuple with lexicographic ordering:

| Tier | Value | Definition |
|---|---|---|
| 4 | Exact match | `entry.label.toLowerCase() === query.toLowerCase()` |
| 3 | Prefix match | `entry.label.toLowerCase().startsWith(query.toLowerCase())` |
| 2 | Substring match | `entry.label.toLowerCase().includes(query.toLowerCase())` |
| 1 | Subsequence match | fzf-style character-by-character subsequence check |
| 0 | No match | excluded from results |

`match_start` is the index where the match begins (prefer earlier matches). `kind_rank` orders kinds: view=3, module=2, file=1, symbol=0 (higher ranks surface first — views are navigation). `label_len` prefers shorter labels (more specific). Final two string fields guarantee determinism when all else ties.

The sort MUST be stable with the complete key. JavaScript's `Array.prototype.sort` is guaranteed stable per ECMAScript 2019+.

**Empty-query state.** When the user opens the palette with no query typed, show: all 6 view entries (always surfaced first), then the 10 highest-PageRank files from the search index. Max 16 entries in empty state. Provides a quick-access list for power users.

**Query echo safety.** The user's raw query is rendered in the results list (for "No results for <query>" and for match highlighting). Query text is rendered via `textContent` only. Match highlighting uses `document.createElement('mark')` around matched characters — NOT `innerHTML` with interpolated query text. Regression test: load palette with query `<img src=x onerror=alert(1)>` and assert the string does not become a DOM tree with an `<img>` node.

**Unicode/bidi safety.** File paths rendered in the palette are scanned for bidi control characters (U+202A–U+202E, U+2066–U+2069). If found, the path is rendered with a `⚠` prefix and the control chars replaced by U+FFFD REPLACEMENT CHARACTER, with a tooltip: `"This path contains bidi control characters"`. NUL bytes (U+0000) cause the entry to be rejected at `build_search_index()` time, with a warn log line.

**Injection safety:** The search index JSON is embedded inline via `<script type="application/json">`. All serialization goes through `escape_script_tag()` (promoted to `pub(crate)` in § 1.2) to handle `</script>` and `<!--` sequences that could escape the tag. This is required for every embedded JSON tag, not just the layout data — search index entries containing file paths with HTML characters would otherwise break the page.

### 1.5 Inspector Panel

Right-side drawer (340px wide) that slides in when a graph node is clicked. Shows:

- **Header:** file/symbol/module name, close button
- **Metrics section:** PageRank, risk score, token count, health sub-scores
- **Flags section:** god file / has dead code / is circular (only if flagged)
- **Actions section:** "Jump to Architecture view" (navigates to `#architecture?file=<path>` for file nodes or `#architecture?module=<prefix>` for module nodes) and "Jump to Risk view" (navigates to `#risk?focus=<path>`).

Target hashes are constructed by URL-encoding the node id with `encodeURIComponent` then concatenating. `href` is set via `setAttribute`, never via template-string concatenation.

**Click behavior when inspector is open:**
- Clicking a different graph node **replaces inspector content** with the new node (no close/reopen flicker).
- Clicking the background (non-node area) **closes** the inspector.
- Clicking inside the inspector itself is a no-op on the inspector state (the click targets the inspector's own widgets).

**Close triggers:** close button, Escape key (see § 1.7 priority rules for the palette/inspector/help-overlay order), click on non-node background, router view-switch (§ 1.3). Cross-references: § 1.7 lists the Escape sequence; § 1.3 lists router-initiated closes.

**State** (open/closed, current node id) lives in `CX.state.inspector` — no localStorage persistence (per-session only).

**Reconciliation with "no shared cross-view selection state":** the inspector's "Jump to" actions navigate via URL hash, and the target view responds to `focus`/`file`/`module` params by panning and highlighting (§ 1.3). This is **hash-based navigation**, not a JS-level shared store — refreshing the page preserves the focus via the URL, but closing and re-opening the file loses it. This distinction matters for v2.2.0 when a real store may be added.

**XSS safety.** All inspector fields use `textContent` — not `innerHTML` — for file paths, symbol names, and module names. Numeric fields use `textContent` with formatted numbers. Links construct `href` via `setAttribute` after `encodeURIComponent` on each path segment.

### 1.6 Theme System

**Two themes:** `dark` (default) and `light`. Toggle in header.

**Icon convention (next-state):** show the icon that represents the state the user will switch TO, not the current state. Sun icon when in dark mode (click → light mode). Moon icon when in light mode (click → dark mode). Matches macOS, VS Code, GitHub. Button `aria-label` updates dynamically: `"Switch to light mode"` / `"Switch to dark mode"`.

**Storage:** `localStorage["cxpak-theme"]` holds `"dark"` | `"light"` | (absent for auto). On first load, if no localStorage value, read `prefers-color-scheme` media query. Theme applied via `data-theme` attribute on `<html>`.

**localStorage validation.** The value is read through a whitelist — anything other than the string literals `"dark"` or `"light"` is discarded and treated as if the key were absent. This is a defense-in-depth guard against localStorage poisoning (e.g. a browser extension writing a malformed value). Applied via `setAttribute('data-theme', …)`, never via template-string HTML.

**localStorage unavailability.** Safari strict mode, private browsing, and some embed contexts throw `DOMException` on any localStorage access (including reads). All localStorage reads and writes in the controller are wrapped in try/catch. On any exception during bootstrap, `CX.state.localStorageAvailable = false` is set once and all subsequent access is skipped silently. The theme falls back to `prefers-color-scheme`; the user can still toggle during the session but the choice does not persist.

**Approach:** CSS attribute-based theming via `[data-theme="light"]` on `<html>` — NOT the CSS `light-dark()` function. Rationale: `light-dark()` requires `color-scheme: light dark` which causes form controls and scrollbars to flip based on OS preference even when we explicitly pin the theme. Attribute-based gives us deterministic control over every element.

**Light palette** is a first-class design, not inverted dark. Keep the same accent hues (blue, green, yellow, red, purple) but shift 15% darker for contrast on off-white #f8f9fc background. Body text #1a1a2e, not pure black.

### 1.7 Keyboard Navigation

| Key | Action |
|---|---|
| Cmd+K / Ctrl+K | Open command palette |
| Escape | Close topmost layer (see priority rule below) |
| ↑ ↓ | Navigate palette results / navigate focused graph nodes |
| Enter | Activate palette result / open inspector for focused node |
| Tab | Cycle through header elements + focus trap in palette when open |
| 1–6 | Switch to view N (dashboard=1, architecture=2, risk=3, flow=4, timeline=5, diff=6) |
| T | Toggle theme |
| / | Alias for Cmd+K |
| ? | Show keyboard shortcut help overlay |

**Escape priority rule.** When multiple layers are open, Escape closes them one at a time in this order: (1) command palette, (2) inspector panel, (3) help overlay. If only one layer is open, Escape closes it. If none are open, Escape is a no-op. Explicit so that users pressing Escape repeatedly reach a known state.

**Focus management** on graph nodes uses the **roving tabindex** pattern: only one node at a time has `tabindex="0"`, all others `tabindex="-1"`. Arrow keys update which node is focused. Container has `role="list"`, each node `role="listitem"`.

**Initial focus on page load.** On `DOMContentLoaded`, `tabindex="0"` is set on the first nav tab (`#nav-dashboard`) so keyboard users tabbing into the page hit the nav first. Graph node roving tabindex is initialized when the first view renders — the first node in reading order receives `tabindex="0"`.

**Screen-reader announcements.** On every view switch (whether triggered by nav tab, number key, or hash change), the controller updates a visually-hidden `<div role="status" aria-live="polite">` with the view name, e.g. `"Switched to Architecture Explorer"`. This is the **only** `aria-live` region in the SPA. Command palette results use `aria-activedescendant` on the input, not a live region, to avoid double-announcement.

**Keyboard shortcut help overlay.** Triggered by `?`. Renders a modal listing the shortcut table above. Not a hidden feature — **in scope for v2.1.0** as a small sub-task (below), superseding any earlier "not building" notes.

### 1.8 Data Freshness Badge

Displayed in the header, right side. Computes from `metadata.generated_at` (ISO 8601 UTC from Rust `chrono::Utc::now()`).

| Age | Display | Class |
|---|---|---|
| < 1 hour | "just now" | `fresh` (muted gray) |
| 1–24 hours | "3h ago" | `fresh` (muted gray) |
| 24–72 hours | "2d ago" | `stale` (amber badge) |
| > 72 hours | "5d ago · copy refresh command" | `old` (red badge) |

Relative time updates every 60 seconds via `setInterval`. The interval handler is paused when `document.hidden` is true (tab not focused) and resumed on `visibilitychange` — avoids background timer wakeups.

**"Copy refresh command" action (old state only):** click on the badge copies `cxpak visual` to clipboard via `navigator.clipboard.writeText()`, shows a 2-second "Copied!" toast.

**Clipboard feature detection.** Older browsers on `file://` context lack `navigator.clipboard`. At bootstrap, the controller tests `typeof navigator.clipboard?.writeText === 'function'`. If unavailable, the badge renders `<code>cxpak visual</code>` inline in the "old" state but **no copy button** — the user reads and types the command manually. This is honest about what the user needs to do without showing a button that silently fails.

**Clock skew caveat.** Relative time is computed client-side using `Date.now() - Date.parse(metadata.generated_at)`. A client whose clock is wrong by >1 hour will display inaccurate relative times. Acknowledged limitation; not worth a separate fix in v2.1.0. Tooltip on the badge always shows the absolute UTC timestamp from `metadata.generated_at` so users can sanity-check.

### 1.9 ARIA Labels in Rust

New field on `LayoutNode`:

```rust
pub struct LayoutNode {
    // ... existing fields ...
    #[serde(default)]
    pub aria_label: String,
}
```

Computed at layout-build time via:

```rust
pub fn build_aria_label(node: &LayoutNode) -> String
```

Rules:
- Always start with `node.label`
- Append node type: "module" | "file" | "symbol" | "group of N items"
- If `risk_score > 0`: append severity phrase (e.g. "high risk (82%)")
- If `token_count > 0`: append token count (e.g. "420 tokens")
- Append flags in order: god file, contains dead code, participates in circular dependency

The `#[serde(default)]` attribute preserves backward compatibility with v2.0.0 cached layouts (they deserialize with `aria_label: ""`).

**DOM insertion safety:** The controller JS must apply the ARIA label via `element.setAttribute('aria-label', node.aria_label)`, never via `innerHTML` or template-string concatenation. `setAttribute` escapes attribute values correctly; string concatenation does not. A file path containing `"` (valid on Linux/macOS) would otherwise break out of the attribute. This is a spec invariant: the controller JS must treat all labels, paths, and symbol names as untrusted data when inserting into the DOM.

**Content allowlist (defense against future expansion).** `build_aria_label` reads ONLY these fields: `node.label`, `node.node_type` (including the `member_ids` slice in `NodeType::Cluster { member_ids }` pattern-match, where only `.len()` is read), `metadata.risk_score`, `metadata.token_count`, `metadata.is_god_file`, `metadata.has_dead_code`, `metadata.is_circular`. Additions outside this allowlist require a spec amendment. Correct Rust pattern:

```rust
match &node.node_type {
    NodeType::Cluster { member_ids } => parts.push(format!("group of {} items", member_ids.len())),
    // ... other variants ...
}
```

This prevents future contributors from innocently embedding source content (file contents, doc-comments, etc.) into ARIA strings, which would leak into screen-reader announcements and browser accessibility trees (surfaces that can be captured by crash reports). Test: grep `build_aria_label` body for any field reads outside the allowlist; fail on mismatch.

### 1.10 Reduced Motion

CSS media query:

```css
@media (prefers-reduced-motion: reduce) {
```

### 1.11 Content-Security-Policy Posture

**Accepted risk.** The architecture requires inlined JS and inlined CSS. Any CSP forbidding `unsafe-inline` would break the dashboard.

- **`file://` context:** browsers do not apply CSP from filesystem URIs. Users opening `cxpak-all.html` directly run with no CSP. This is accepted and unavoidable for the self-contained HTML contract.
- **`cxpak serve` context:** the server sets `Content-Security-Policy: default-src 'self'; script-src 'unsafe-inline' 'self'; style-src 'unsafe-inline' 'self'; img-src 'self' data:; connect-src 'self'; frame-ancestors 'none'`. This permits the inlined content while blocking external resources, data exfiltration via third-party connections, and framing attacks.
- **No `eval`:** the controller JS must not use `eval()`, `new Function()`, or `setTimeout` / `setInterval` with string arguments. Test: grep the controller asset for these patterns.

#### 2.1.1 Focus / files / symbol / target semantics

**Two normalizers, not one.** Path-like parameters (file paths, module paths) use a strict path whitelist. Symbol-like parameters (`V1DataFlowParams.symbol`, `V1CallGraphParams.target`) may legitimately contain `<>:()` (e.g. `Vec<String>`, `std::vector<int>`, `impl Trait`) and need a broader allowlist:

```rust
// Shared error helper (defined once per § 2.1 error response shape).
// Returns (StatusCode, Json<Value>) for the axum handler.
type V1Result<T> = Result<T, (axum::http::StatusCode, axum::Json<serde_json::Value>)>;

fn v1_error(status: StatusCode, code: &'static str, msg: impl Into<String>) -> (StatusCode, Json<Value>) {
    (status, Json(json!({"error": code, "message": msg.into()})))
}

fn normalize_path_param(value: &str) -> V1Result<String> {
    if value.len() > 1024 { return Err(v1_error(StatusCode::BAD_REQUEST, "param_too_long", "path exceeds 1024 chars")); }
    if value.contains('\0') || value.contains('\\') { return Err(v1_error(StatusCode::BAD_REQUEST, "invalid_param", "illegal character")); }
    // Reject `..` only as a standalone path segment (not as substring in filenames like `foo..bar.txt`).
    if value.split('/').any(|seg| seg == "..") { return Err(v1_error(StatusCode::BAD_REQUEST, "invalid_param", "path traversal segment")); }
    if value.starts_with('/') { return Err(v1_error(StatusCode::BAD_REQUEST, "invalid_param", "absolute path")); }
    // Whitelist: alphanumerics, underscore, hyphen, dot, forward slash only.
    if !value.chars().all(|c| c.is_ascii_alphanumeric() || matches!(c, '_' | '-' | '.' | '/')) {
        return Err(v1_error(StatusCode::BAD_REQUEST, "invalid_param", "illegal character class"));
    }
    Ok(value.to_string())
}

fn normalize_symbol_param(value: &str) -> V1Result<String> {
    if value.len() > 512 { return Err(v1_error(StatusCode::BAD_REQUEST, "param_too_long", "symbol exceeds 512 chars")); }
    if value.contains('\0') { return Err(v1_error(StatusCode::BAD_REQUEST, "invalid_param", "null byte")); }
    // Symbol names in 42 supported languages can contain a broad set of chars.
    // Reject only control chars and characters known to be shell/path-injection vectors.
    if value.chars().any(|c| c.is_control() || matches!(c, '/' | '\\' | '`' | '$' | ';' | '|')) {
        return Err(v1_error(StatusCode::BAD_REQUEST, "invalid_param", "illegal character"));
    }
    Ok(value.to_string())
}
```

**Cap justification:** 1024-char path limit chosen to accommodate deep monorepo paths (observed: Bazel `bazel-out/k8-fastbuild/bin/.../foo.pb.h` hits ~400 chars, deep Kotlin Multiplatform paths reach ~600). 1024 is comfortable headroom. 512-char symbol limit matches real-world upper bound for generic-heavy C++/Rust signatures. 100 elements for `Vec<String>` matches the practical "files changed in a PR" upper bound. 10 depth cap for predict/data_flow — beyond 10 the blast radius becomes the entire codebase, providing no insight.

**Empty-string normalization.** Before either normalizer runs, handlers convert `Some("")` to `None` (empty string is semantically "no filter"). This removes the asymmetry where `focus: ""` 400s but `focus: null` passes through. Applied centrally in a pre-normalization pass on all `Option<String>` fields.

**Vec input caps.** `Vec<String>` inputs are capped at **100 elements** before per-element normalization (to bound work per request).

**Depth caps.** `V1PredictParams.depth` and `V1DataFlowParams.depth` default to 3 (matching existing non-v1 handlers) and cap at **10** (returning `400 "depth_exceeds_max"` when higher). Without a cap, `depth: usize::MAX` causes BFS to walk the full transitive graph.

**Applied to:** every element of `files` (path normalizer), the `focus` field (path), `workspace` (path), `target` in call_graph (path; functions/methods in call graph are file-qualified), and `symbol` in data_flow (symbol normalizer).

#### 2.1.2 Authentication

The bearer-token middleware from v2.0.0 is preserved unchanged. Additions in v2.1.0:

- **Default bind behavior:** when `--token` is unset, `cxpak serve` binds to `127.0.0.1` only and logs a warning: `"warn: serving without --token; binding to loopback only"`. Binding to any **non-loopback** address (including `0.0.0.0`, `[::]`, LAN addresses like `192.168.x.x`, public IPs) without `--token` is rejected with an error at startup. Loopback check: `SocketAddr::ip().is_loopback()` on the parsed bind address — covers IPv4 `127.0.0.0/8` and IPv6 `::1` via stdlib.
- **Header-only auth.** Tokens are accepted ONLY via `Authorization: Bearer <token>` header. Query-param auth (`?token=...`) and body-field auth are explicitly rejected; tokens in URLs leak into access logs, browser history, and referer headers. Test: a request with `?token=...` returns 401 even if the token is correct.
- **Timing-safe comparison:** existing timing-safe comparison in v2.0.0 is preserved. Regression test confirms.
- **Logging:** tokens MUST NOT appear in `tracing` spans, access logs, error response bodies, or CLI output.
  - **Test approach (since tower's `oneshot` runs in-process):** implement as a subprocess integration test in the style of `tests/serve_test.rs` — spawn `cxpak serve --token <distinctive-token>` as a child process, make HTTP requests via `TcpStream`, capture stderr via `Child::stderr`, grep for the token substring.
- **Auth applies to all `/v1/*`:** including the 9 newly-wired endpoints. Existing middleware covers them automatically via route-path match.

## Data Integrity Contracts

**The hard invariant of this release: every number displayed in the SPA must come from the same intelligence function that powers the CLI and MCP tools.** No duplicated computation. No off-by-one counts. No rounding divergence.

### Contracts that must hold

1. **Health score.** `dashboard.health.composite` in embedded JSON **must equal** `compute_health(&idx).composite` after a JSON serialization round-trip. Test comparison: compute the Rust value, run it through `serde_json::to_value(..).and_then(|v| v.as_f64())`, and `.to_bits()` against the value extracted the same way from the embedded HTML. Because both sides go through identical serialization, drift at the Rust arithmetic layer is detected. Pure `.to_bits()` against the original Rust value is documented as insufficient because JSON serialization can change the bit pattern by 1 ULP for boundary values.

2. **Top risks.** `dashboard.risks.top_risks` **must be the first N entries** of `compute_risk_ranking(&idx)` where `N = min(5, total_risks)`, in the same order, with each `risk_score` compared via the JSON round-trip method described in Contract 1. Every `path` string matches exactly.

3. **Risk ranking tie-breaking (required addition to `compute_risk_ranking`).** Two entries with equal `risk_score` bits sort by `path` ascending as a secondary key. The sort comparator in `src/intelligence/risk.rs` must be updated to:

```rust
use std::cmp::Ordering;
risks.sort_by(|a, b| {
    b.risk_score
        .partial_cmp(&a.risk_score)
        .unwrap_or(Ordering::Equal)
        .then_with(|| a.path.cmp(&b.path))
});
```

Spec change because the current deterministic behavior is emergent from input order and brittle to unrelated refactors. Dedicated regression test: `risk_ranking_ties_break_by_path_ascending` constructs two entries with identical `churn_30d`, `blast_radius`, `has_tests`, and `coverage` (all inputs into risk_score), verifies they get identical f64 bits, then verifies the sort places them by lexicographic path.

4. **Architecture modules.** `architecture.level1.nodes.len()` **must equal** `build_architecture_map(&idx, 2).modules.len()` (after cognitive-limit clustering, cluster node counts as 1).

5. **Search index completeness.**
   - Every entry in `idx.files` produces a `kind: "file"` entry.
     - "Parse failed" means `file.parse_result.is_none()` — the parser errored on this file. Detail: `"{language} · parse error"` (or `"unknown · parse error"` if language is absent).
     - "No symbols extracted" means `parse_result` is `Some(_)` but `parse_result.symbols.is_empty()` — distinct from parse failure. Detail: `"{language} · {token_count} tokens"` — treated as a normal file, no special marker.
   - Every **public** symbol in every parsed file (`visibility == Visibility::Public`) produces a `kind: "symbol"` entry. Private symbols are excluded.
   - Every unique module prefix (first two path segments; see § 1.4) produces a `kind: "module"` entry.
   - All 6 view entries always present.
   - **Cap carve-out:** when `build_search_index()` emits more than 20,000 entries, the list is truncated per § 1.4. Contract 5's completeness requirement then applies only to: all 6 views, all module prefixes, and the top-ranked (by PageRank) files and symbols fitting under the cap. The truncation is reported via a warn log line; tests asserting completeness on large fixture repos MUST account for this (either use a small-enough fixture, or assert both "under cap → full completeness" and "over cap → warn + partial completeness" as separate cases).

6. **Search index determinism.** Running `build_search_index()` twice on the same index produces identical `Vec<SearchEntry>` byte-for-byte after serialization. Entries sorted by `(kind, label, context)` — the `context` tertiary key resolves collisions between symbols with identical names in different files.

7. **Node count match.** `metadata.node_count` **must equal** `idx.total_files`.

8. **Edge count match.** `metadata.edge_count` **must equal** `idx.graph.edges.values().map(|v| v.len()).sum::<usize>()`, computed once in a shared helper and used by both the SPA renderer and the existing per-view renderers — not recomputed independently in each call site.

9. **Cross-channel risk consistency.** For a given index with NO focus filter applied on any channel:
   - `spa.dashboard.risks.top_risks` = first 5 entries of `compute_risk_ranking(&idx)`
   - `POST /v1/risks` (body `{}` or `{"focus": null}`) response body = `{"risks": compute_risk_ranking(&idx)}`
   - `cxpak_risks` MCP tool output = `compute_risk_ranking(&idx)` (serialized)

   Per-entry comparison uses the JSON round-trip method from Contract 1; per-list comparison uses length-then-pairwise. **When `focus` IS set on any channel, that channel applies its documented filter (§ 2.1.1) and the cross-channel equality does NOT hold** — this is expected divergence, not a bug.

   The existing filter in `src/visual/render.rs:1273` (`.filter(|e| e.risk_score >= 0.05)` in `build_dashboard_data`) **is explicitly REMOVED in v2.1.0**. The existing test at `src/visual/render.rs:3796` that asserts the filter is correct **is updated to assert the opposite**: that `top_risks` is the first 5 of the unfiltered ranking. This is called out to prevent the spec-vs-test contradiction that would otherwise occur at implementation time.

   **Scope note:** Contract 9 applies ONLY to the `top_risks` quadrant (first five of the ranking). The **separate** filter at `src/visual/render.rs:1333` (`.filter(|e| e.risk_score > 0.8).take(3)`) powers the "Alerts" quadrant and is unchanged — alerts intentionally show only critical files, a different UI contract from top_risks. Contract 9 does not cover the alerts quadrant.

   No LSP method covers risk ranking directly (LSP's `cxpak/securitySurface` calls a different function), so LSP is not part of this specific contract.

10. **Cross-channel health consistency.** For a given index, the **`composite` f64** field must agree via `.to_bits()` after JSON round-trip across:
    - `spa.dashboard.health.composite`
    - `POST /v1/health` response `composite` field
    - `cxpak_health` MCP tool output `composite` field
    - LSP `cxpak/health` method output `composite` field (existing pre-wired method from v1.6.0)

    This is a scalar comparison, not byte-equality of whole response bodies (which differ in envelope shape across channels).

11. **Cross-channel architecture consistency.** For `build_architecture_map`, the **module count** and **module prefix list** (as a sorted set of strings) must agree across:
    - `spa.architecture.level1.nodes` length (excluding cluster nodes from the 7±2 enforcement) and the set of module prefix strings extracted from those nodes
    - `POST /v1/architecture` response `modules.len()` and set of `modules[].prefix`
    - `cxpak_architecture` MCP tool output `modules.len()` and set of `modules[].prefix`

    **No LSP method covers architecture** (there is no `cxpak/architecture` method in v1.6.0 or v2.1.0). LSP is not part of this contract.

    **Per-node aria_label divergence is expected.** The SPA's `LayoutNode` has an `aria_label` field (§ 1.9); the v1/MCP `ModuleNode` does not. Contract 11 compares module-count and module-prefix-set only — not whole-node byte-equality. The matrix in § Testing Strategy reflects this with field-level comparison rather than struct-level.

12. **Health score formatting consistency.** The raw f64 is embedded in JSON. The dashboard renders it via a shared format helper `CX.format.score(x: number): string` (one JS function, defined once in the controller bootstrap section) so that `compute_health().composite = 9.95` renders the same way in the header badge, the gauge, and the freshness tooltip. No scattered `.toFixed(1)` calls for score values. Contract applies specifically to **health/risk scores**; integer-like formatters (`Math.floor`, `String(Math.round(x))`) remain allowed for non-score values like hours and days in the freshness badge. Test: grep the controller asset for `toFixed` — every match must appear inside the body of `CX.format.score`, not elsewhere.

13. **HTML attribute safety.** All DOM insertions of user-derived strings (file paths, symbol names, ARIA labels, router param echoes) go through `setAttribute()` or `textContent`, never `innerHTML` or template-string attribute concatenation.

**Test implementation:** the grep asserts are implemented in a Rust integration test `tests/controller_dom_safety.rs` that reads `assets/cxpak-spa-controller.js` via `include_str!`. The test FIRST asserts the file is non-empty (`assert!(controller.len() > 1000, "controller asset must be populated before this test is meaningful")`) to prevent vacuous passes when the asset is absent or empty. Then it runs the regex checks:

- `\binnerHTML\s*[+]?=` — zero matches required (innerHTML writes are forbidden).
- `d3\.select(?:All)?\([^)]+\)\.html\s*\(` — D3 `.html()` method calls. Each match must be immediately followed (within the next 2 lines) by a code comment `// safe: static markup, no user input`. Unannotated matches fail the test.
- `outerHTML\s*[+]?=` — zero matches required.
- `document\.write\s*\(` — zero matches required.

The regex uses `\b` / `[^)]+` to avoid false matches on helper names like `fooHTML` or unrelated `.html(` calls on non-D3 objects.

### How integrity tests enforce the contracts

Integration tests call `render_spa()`, then **parse the embedded JSON from the specific `<script id="cxpak-*-data">` tag**, then compare against direct calls to the intelligence functions. The comparison extracts numeric fields from JSON — it does **not** search the raw HTML string for formatted values (which would miss third-decimal drift and be vulnerable to coincidental substring matches).

Marker extraction uses `.expect()`, not `if let Some()`. If a data tag is missing, the test fails loudly. No silent passes.

Realistic test indices contain at least **10 files with cross-module imports** so that intelligence functions return meaningful (non-empty) data. Single-file indices are reserved for testing empty-state behavior.

## Testing Strategy

### Unit test coverage target: 100% of new Rust code

Measured by `cargo tarpaulin --all-features`. Excludes:
- D3 bundle and CSS (inlined assets, no Rust)
- JS controller modules (treated as inlined strings; integration-tested via HTML output parsing)

Each new function requires:
- Happy path test
- At least one edge case (empty input, single item, boundary condition)
- At least one error path where applicable

### Strict TDD: red-green-refactor (Rust code)

Every **Rust** implementation task follows:
1. Write the failing test (must produce the error that falsifies the current code; `assert!(x <= 5)` is not acceptable when current code returns 3 — the test passes)
2. Run and confirm failure with specific expected error
3. Implement minimal code
4. Run and confirm pass
5. Refactor if needed
6. Commit

A test that cannot fail the pre-implementation code is a plan defect and must be fixed before proceeding.

### JS controller: verified by grep + golden fixture, NOT TDD

`assets/cxpak-spa-controller.js` is verified by three mechanisms:

1. **Rust-side grep assertions** (Contracts 12, 13 + the DOM-safety test) that parse the asset file as a string.
2. **HTML-output integration tests** that render the SPA, parse the embedded JSON data tags, and assert data integrity contracts.
3. **Cross-process golden fixture** (§ Determinism test) comparing byte-level output against a committed snapshot.

This is **regression testing**, not TDD — the controller is written-then-verified, not test-first. Rationale: unit-testing JS without a headless browser requires jsdom + node, which conflicts with the "one Rust binary, no npm" principle. The controller's logical surface is narrow (routing, palette, inspector, theme) and each piece is either grep-visible or measurable via output parsing. Real DOM-level bugs that slip through grep + output parsing are caught by the manual smoke test (Success Criterion 11).

**Acknowledged risk:** a logic bug that neither changes grep-visible patterns nor produces wrong JSON output could survive CI and be caught only at smoke test. v2.2.0 may add headless browser tests; v2.1.0 does not.

### E2E data integrity tests

Located in `tests/spa_e2e.rs`. For each contract in § Data Integrity Contracts:

```rust
// Pattern: extract, parse, compare numerically
let marker = r#"id="cxpak-dashboard-data" type="application/json">"#;
let start = html.find(marker).expect("dashboard data must exist");
let json_start = start + marker.len();
let json_end = html[json_start..].find("</script>").expect("data tag must close") + json_start;
let json_str = &html[json_start..json_end];

```

Uses `to_bits()` for exact f64 comparison (catches NaN, -0.0 vs +0.0, precision drift). Never uses `"{:.1}"` string matching against raw HTML.

### Determinism test — cross-process

The deterministic-output test is **not** two calls in one process (HashMap seed is stable within a process and would pass trivially). Instead:

1. The golden fixture `tests/snapshots/spa_golden.html` is **committed to the repo**. A fresh checkout has it available without bootstrapping.
2. A separate test loads the golden fixture and diffs against a fresh `render_spa()` call using the fixture repo at `tests/fixtures/determinism_repo/`.
3. **Updates to the fixture are intentional and local-only**: `UPDATE_SNAPSHOTS=1 cargo test` regenerates the fixture. **CI must NOT set this variable** — CI runs with `UPDATE_SNAPSHOTS` unset, so any drift fails the test.
4. **First-run bootstrap:** if the golden file is missing entirely (e.g. immediately after the snapshot was added to the plan but before it was committed), the test **fails** with a clear message: `"run UPDATE_SNAPSHOTS=1 cargo test to bootstrap tests/snapshots/spa_golden.html"`. No automatic creation — committing a snapshot must be a deliberate action.
5. **Redaction:** fields with unavoidable non-determinism are redacted to fixed placeholders before comparison. The redaction list is exhaustive and maintained in one place (`tests/support/redact.rs`):
   - `metadata.generated_at` (RFC 3339 timestamp)
   - `metadata.cxpak_version` (version string)
   - `DriftSnapshot.timestamp` (RFC 3339, present on every snapshot — verified at `src/intelligence/drift.rs:19`)
   - `BaselineComparison.baseline_date` (RFC 3339 — verified at `src/intelligence/drift.rs:43`)
   - Any future timestamp field: ADD TO the redaction helper, don't work around per-test.

This catches non-determinism that only manifests across processes (e.g., HashMap iteration order persisted into the JSON output).

### Integration tests for platform completeness

- `tests/v1_api_wired.rs`: every `/v1/*` endpoint POSTed with realistic params, assert response is non-stub, assert schema matches the intelligence function's return type.
- `tests/lsp_methods_wired.rs`: every custom method called with realistic params, assert response is non-stub, assert structure is consistent with the intelligence function.
- `tests/mcp_slug_validation.rs`: pass malicious `type` params to `cxpak_visual` MCP tool — verify error before any file is written.

**Regression framing note:** The existing 2,436 tests do **not** assert stub behavior for the 9 v1 endpoints or 11 LSP methods being wired (confirmed by grep against `tests/api_v1_integration.rs` and `tests/lsp_integration.rs`). So "all existing tests still pass" is a weak regression gate for Pillar 2 — it catches unrelated breakage only. The actual gate is the new integration tests above. This is why these tests are **required** before wiring any handler (TDD red phase), not optional.

**Stub-detection matcher** uses a canonical predicate that matches the v2.0.0 stub shapes precisely:

```rust
fn is_stub(body: &serde_json::Value) -> bool {
    if let Some(status) = body.get("status").and_then(|s| s.as_str()) {
        return status == "not_implemented" || status == "available";
    }
    false
}
```

Not a brittle substring search.

### Edge-case test requirements (checklist — each must have a named test)

**Zero / boundary values:**
- `spa_renders_with_zero_files_index` — build an empty index, render SPA, assert no panic, all views render empty-state cards, palette opens but shows only the 6 view entries.
- `v1_risks_zero_files_returns_empty_envelope` — zero-file index → `POST /v1/risks` returns 200 with `{"risks": []}`, NOT 204 or 500.
- `health_gauge_renders_zero_composite` — force composite=0.0, assert HTML contains `"0.0"` in the gauge JSON, not missing or hidden.
- `health_gauge_renders_max_composite` — force composite=10.0, assert HTML contains `"10.0"`.
- `risk_filter_boundary_inclusive` — index with one risk at exactly score `0.05`, render dashboard, assert entry is present (the spec removes the 0.05 filter in Contract 9).
- `compute_risk_ranking_ties_break_by_path` — two entries with identical bit pattern → sort by path ascending.

**Parse-failure distinction (Contract 5):**
- `search_index_parse_failed_file_has_error_marker` — file with `parse_result: None` produces entry with detail containing `"parse error"`.
- `search_index_empty_symbols_file_has_no_error_marker` — file with `parse_result: Some(ParseResult { symbols: vec![], ... })` produces entry with regular detail (no parse error text).

**Symbol-character handling:**
- `v1_data_flow_accepts_angle_brackets_in_symbol` — `symbol: "Vec<String>"` is accepted (normalize_symbol_param allows `<>`).
- `v1_data_flow_rejects_path_separator_in_symbol` — `symbol: "../etc/passwd"` returns 400 (symbol normalizer rejects `/` and `\`).
- `search_index_handles_utf8_symbol_names` — symbol named `módulo` (valid Rust identifier in some languages' bindings) round-trips correctly.
- `palette_rejects_nul_byte_in_entry` — a file path containing `\0` is dropped from the search index at build time.
- `palette_flags_bidi_override_in_path` — file path containing U+202E renders with `⚠` prefix and replacement chars.

**Browser-API fallbacks (grep-based assertions against the controller asset):**
- `controller_guards_localStorage_in_try_catch` — grep `localStorage` and assert every match is preceded by or inside a `try` block within 10 lines.
- `controller_feature_detects_clipboard` — grep for `navigator.clipboard` and assert it appears inside a `typeof … === 'function'` check.
- `controller_pauses_freshness_interval` — grep for both `document.hidden` and `visibilitychange` — both must appear (ensures the pause-on-hidden logic is wired).

**Escape priority (no JS test infra — use grep + structural assertions):**
- `controller_escape_handler_defined` — grep for `case 'Escape'` or `key === 'Escape'` in the controller; must appear.
- `controller_escape_closes_palette_first` — structural: the escape handler's first branch must reference `paletteOpen` before `inspectorOpen`. Test parses the handler body and asserts the order of conditional checks.

**MCP error response shape:**
- `mcp_visual_invalid_type_returns_jsonrpc_error` — pass `type: "not_a_view"` to `cxpak_visual`. Response is a JSON-RPC error object with `code` (e.g. -32602 for invalid params) and `message: "invalid_type: not_a_view"`.
- `mcp_visual_missing_symbol_on_flow_returns_error` — `type: "flow"`, no `symbol`. Response is JSON-RPC error with `code: -32602, message: "missing_required_param: symbol"`.
- `mcp_visual_slug_traversal_rejected` — `type: "../etc/passwd"`. Response is JSON-RPC error; `.cxpak/visual/` directory contents unchanged.

### Browser-based E2E tests: explicitly deferred

We do not run headless browser tests in v2.1.0. Rationale: adding Chromium infrastructure to CI doubles build time and complicates the "one Rust binary" value proposition. Instead, we verify:

1. The HTML opens in a browser without console errors (manual smoke test during release)
2. Every JS entry point is unit-tested by its Rust caller (data passed to it is correct)
3. The view controller JS is pure enough that logic bugs show up in the data-shape assertions

v2.2.0 will add optional Playwright tests gated behind an environment variable.

## Success Criteria

This release ships when:

1. **All 2,436 existing tests pass.** Plus ~80 new tests across unit/integration/E2E.
2. **Coverage ≥ 90% overall, 100% on new Rust code** (tarpaulin report checked in CI). JS coverage is not measured in v2.1.0; the controller is integration-tested via the cross-channel matrix and DOM assertions.
3. **Stub detection test passes.** `tests/v1_api_wired.rs` uses the canonical `is_stub(&Value)` matcher. Every `/v1/*` endpoint returns non-stub. `tests/lsp_methods_wired.rs` covers **all 14 LSP custom methods** — the 11 previously stubbed plus the 3 pre-wired (which must continue to return non-stub responses; backwards-compatibility assertion).
4. **Cross-channel consistency matrix passes.** `tests/cross_channel_consistency.rs` covers every cell in the § Testing Strategy matrix. Surfaces that don't expose a given function (dashed cells) have no test and no code path — attempting to test them is a spec error.
5. **`cxpak visual` produces a valid SPA HTML** that passes all 13 § Data Integrity Contracts.
6. **Cross-process determinism test passes.** Golden fixture at `tests/snapshots/spa_golden.html` (committed) diffed against fresh render. Updates via `UPDATE_SNAPSHOTS=1 cargo test` are **forbidden in CI** (env var unset in CI config).
7. **Size budget test passes.** Running `render_spa()` on the cxpak repo itself produces HTML < 10 MiB. Hard fail at 20 MiB.
8. **Injection safety tests pass.** (a) `render_spa()` on an index containing a file path literally `</script><img src=x onerror=alert(1)>.rs` produces HTML where the JSON-embed is safely escaped. (b) Load `#architecture?focus=<script>alert(1)</script>` in the SPA and assert the script string does not appear in the DOM. (c) Pass a slug `"../etc/passwd"` to `cxpak_visual` MCP — assert error before any file write.
9. **Version 2.1.0** across all four version files, `Cargo.lock` regenerated.
10. **Zero TODO/FIXME/unimplemented!()/todo!() in new code.** Enforced by clippy lints and a grep check in CI.
11. **Manual browser smoke test** (explicitly manual, not part of the automated CI gate — Criteria 1–10 form the automated gate). **Scheduled after Criteria 1–10 pass and AFTER the golden fixture is regenerated (Criterion 6) following the version bump (Criterion 9). Before the git tag.** Procedure documented in `docs/release/v2.1.0-smoke.md` (written as part of the release task). Exact checklist:
   - Open `cxpak-all.html` generated from the cxpak repo itself in Chrome, Firefox, Safari (latest + one version prior).
   - DevTools console shows zero errors (warnings OK). Network tab shows zero requests (everything inlined).
   - All 6 nav tabs switch views without reload.
   - Cmd+K opens palette; type "health"; click result; lands on dashboard with focus highlighting.
   - Theme toggle flips colors; localStorage persists across page reload.
   - Click a node on any graph → inspector opens on the right with correct metrics.
   - Press `?` — help overlay lists every shortcut from § 1.7.
   - Close tab; reopen; theme preference preserved.

   The release is held on any smoke-test failure. The smoke test is a gate, but a manual one — by design, not a CI automated test.

## What We Are Not Building

State these explicitly so the implementation plan stays scoped.

**Note on "v2.2.0" labels below:** these are deferral markers, not commitments. When an item is labeled v2.2.0, it means "not in v2.1.0, possibly in a future release if user demand exists." The v2.2.0 scope is not designed yet and is not part of this spec.

- **No new intelligence signals.** Every number in the SPA already exists.
- **No WASM plugin execution.** v2.2.0.
- **No browser E2E automation.** v2.2.0.
- **No responsive layout for mobile.** v2.2.0+ or never.
- **No "Copy as Mermaid" in-browser button.** Rust-side mermaid export already works.
- **No architecture drill-down animation.** Polish pass, not core.
- **No JS-level shared cross-view selection state.** Clicking a file in Risk doesn't automatically highlight it in Architecture via a shared store. But **hash-based navigation** (see § 1.5 inspector Jump actions) allows users to move between views while preserving focus via the URL. A full shared store could be added in v2.2.0 if user demand exists.
- **No onboarding overlay.** The command palette is the onboarding.
