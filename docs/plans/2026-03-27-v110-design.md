# cxpak v1.1.0 Design: Repository DNA

## Summary

Evidence-based convention extraction and verification. cxpak analyzes the codebase's actual patterns — naming, imports, error handling, architecture, testing, visibility, function shape, git health — and produces a quantified convention profile. Every finding is backed by counts, percentages, and exceptions. A ~1000 token DNA section is included in every `auto_context` call. A new `cxpak_verify` tool checks code changes against the profile, reporting violations only in changed lines. Principle: report what IS, never what SHOULD BE.

## Architecture

New module: `src/conventions/`
- `mod.rs` — public types (`ConventionProfile`, `PatternObservation`, `PatternStrength`), profile builder
- `naming.rs` — function/type/file/constant naming pattern extraction
- `imports.rs` — import style, grouping, re-export pattern extraction
- `errors.rs` — error return types, propagation style, unwrap detection
- `deps.rs` — dependency direction between directories, layering detection
- `testing.rs` — coverage per directory, mock detection, test naming, density
- `visibility.rs` — public/private distribution, doc comment coverage
- `functions.rs` — function length stats per directory
- `git_health.rs` — churn (30d + 180d), bug-fix density, revert detection
- `render.rs` — render DNA section (~1000 tokens compact markdown)
- `verify.rs` — diff scoping, convention checking, violation + suggestion generation

Built at index time, stored on `CodebaseIndex` as `pub conventions: ConventionProfile`. Incrementally updated when the file watcher detects changes — only re-extract patterns from changed files, merge delta into cached profile.

**Build-time dependency on repo path:** Convention extraction for `git_health` needs a `repo_path` to call `git2`. This is NOT passed to `CodebaseIndex::build()`. Instead, conventions are populated AFTER `build()` in a separate step, same pattern as how `schema` and `graph` are handled — the `build_index()` function in `serve.rs` (which has `repo_path`) calls `build_convention_profile(&index, repo_path)` after index construction and assigns `index.conventions = profile`. For CLI commands, the repo path is available from the command arguments.

## Core Types

```rust
pub struct ConventionProfile {
    pub naming: NamingConventions,
    pub imports: ImportConventions,
    pub errors: ErrorConventions,
    pub dependencies: DependencyConventions,
    pub testing: TestingConventions,
    pub visibility: VisibilityConventions,
    pub functions: FunctionConventions,
    pub git_health: GitHealthProfile,
}

pub struct PatternObservation {
    pub name: String,
    pub dominant: String,
    pub count: usize,
    pub total: usize,
    pub percentage: f64,
    pub strength: PatternStrength,
    pub exceptions: Vec<String>,
}

#[derive(Debug, Clone, PartialEq, Eq, Serialize, Deserialize)]
pub enum PatternStrength {
    Convention,  // ≥90% consistency
    Trend,       // 70-89% consistency
    Mixed,       // 50-69% consistency
    // <50% = not reported
}
```

Each category struct has typed fields for known patterns plus `pub additional: Vec<PatternObservation>` for dynamic expansion.

### Strength Thresholds

| Strength | Threshold | Verify Severity | Meaning |
|---|---|---|---|
| Convention | ≥90% | `high` | This is how the project does it. Deviations are almost certainly wrong. |
| Trend | 70-89% | `medium` | Most code follows this pattern. Deviations should have a reason. |
| Mixed | 50-69% | `low` | Slight majority. Informational only. |
| Below 50% | — | not reported | No dominant pattern. |

## Feature 1: Pattern Extraction

### Extraction Sources

| Category | Pattern | Data Source | Confidence |
|---|---|---|---|
| **Naming** | Function naming style (snake/camel/Pascal) | `Symbol.name` where kind = Function/Method | 100% |
| | Type naming style | `Symbol.name` where kind = Struct/Class/Enum/Type/Interface/Trait | 100% |
| | File naming style | `IndexedFile.relative_path` filename portion | 100% |
| | Constant naming style | `Symbol.name` where kind = Constant | 100% |
| **Imports** | Absolute vs relative | `Import.source` — check for `./` `../` (relative) vs module paths | 100% |

| | Re-exports | Check `mod.rs` files for `pub use` patterns | 95% |

| | Error type per module | `Symbol.name` where kind = Enum/Struct, name contains "Error"/"Err" | 95% |
| **Dependencies** | Direction between directories | `DependencyGraph` — count edges per directory pair | 100% |

| | Churn trend | Compare 30d vs 180d — hot/stabilized/chronic/cold | 100% |

### Incremental Update

When the file watcher triggers a file change:

1. Re-parse the changed file (already happens in existing incremental update)
2. For naming/imports/errors/visibility/functions: re-extract patterns from the changed file's new `ParseResult`
3. Update counts: subtract old file's contribution, add new file's contribution
4. Recompute percentages and strength labels
5. Git health: not updated incrementally (requires `git log`). Recomputed on `cxpak_verify` calls (since verify references git history) and cached with a 60-second TTL. Within the TTL, the cached git health is reused. After TTL expires, the next verify or conventions call triggers a refresh.

This is O(changed files) per update — no full re-scan needed.

**Per-file contribution tracking:** Each category stores a `HashMap<String, FileContribution>` mapping file paths to their individual pattern counts. When a file changes: look up old contribution, subtract from aggregates, compute new contribution, add to aggregates, update the map. This is a small memory overhead (~100 bytes per file per category) but enables true incremental updates. Hooks into the existing `apply_incremental_update` path in `watch.rs`.

## Feature 2: `cxpak_verify` MCP Tool

### Schema

```json
{
  "name": "cxpak_verify",
  "description": "Verify code changes against the codebase's observed conventions. Reports deviations with evidence, severity, and suggested fixes. Only flags violations in changed lines, not pre-existing debt.",
  "inputSchema": {
    "type": "object",
    "properties": {
      "ref": {
        "type": "string",
        "description": "Git ref to diff against (default: auto-detect uncommitted changes vs HEAD)"
      },
      "focus": {
        "type": "string",
        "description": "Path prefix to scope verification"
      }
    }
  }
}
```

### Response Format

```json
{
  "files_checked": 3,
  "lines_checked": 147,
  "violations": [
    {
      "severity": "high",
      "category": "error_handling",
      "location": "src/auth/jwt.rs:42",
      "message": ".unwrap() used in production code",
      "evidence": {
        "dominant_pattern": "Result<T, AppError> with ? propagation",
        "count": "147/150 functions (98%)",
        "strength": "convention",
        "history": "3 commits reverted unwrap() usage in this module"
      },
      "suggestion": "Replace .unwrap() with .map_err(|e| AppError::JwtValidation(e))?"
    }
  ],
  "passed": [
    "imports: absolute paths, correctly grouped (3/3 files)",
    "architecture: no reverse dependency violations",
    "testing: test file exists for modified module"
  ],
  "summary": {
    "high": 2,
    "medium": 1,
    "low": 0
  }
}
```

### Implementation

1. **Determine diff scope (via `git2`, not CLI):**
   - No args → `git2::Repository::diff_index_to_workdir()` for uncommitted changes vs HEAD
   - `ref` provided → `git2::Repository::diff_tree_to_tree()` between ref and HEAD
   - `focus` → filter diff deltas to paths under prefix
   - Uses `git2` APIs exclusively (already a dependency). No shelling out to `git` CLI. This is testable without a git installation.

2. **Extract changed lines:** parse the unified diff to identify added/modified line number ranges per file. Only these lines are checked.

3. **Re-parse changed files:** get fresh `ParseResult` for each modified file.

4. **Check NEW symbols/code against each convention category:**
   - New function names → naming pattern
   - New function signatures → error return type pattern
   - New function bodies → unwrap detection, import style
   - New files → dependency direction, file naming
   - New imports → import style, grouping
   - New test files → test naming, mock usage

5. **Skip pre-existing code:** if a symbol existed before the diff and its name/signature/body hasn't changed, don't check it. Only check symbols that are new or modified.

6. **Generate violations with evidence:** each violation cites the convention's count, percentage, strength, and git history where relevant.

7. **Generate suggestions where deterministic:**

| Convention | Suggestion | Deterministic? |
|---|---|---|
| Naming (wrong case) | Rename to correct case | Yes |
| `.unwrap()` in production | Replace with `?` + error mapping | Yes (pattern observable) |
| Import style (relative→absolute) | Rewrite import path | Yes |
| Missing doc comment | Add `///` stub | Yes |
| Public where trend is private | `suggestion: null` | No (requires judgment) |
| Architecture violation | `suggestion: null` | No (requires judgment) |

### Severity Mapping

| Pattern Strength | Verify Severity | Meaning |
|---|---|---|
| Convention (≥90%) | `high` | Almost certainly a mistake |
| Trend (70-89%) | `medium` | Should have a reason |
| Mixed (50-69%) | `low` | Informational |

## Feature 3: `cxpak_conventions` MCP Tool

### Schema

```json
{
  "name": "cxpak_conventions",
  "description": "Return the full convention profile for the codebase. Shows all detected patterns with counts, percentages, strength labels, and exceptions.",
  "inputSchema": {
    "type": "object",
    "properties": {
      "category": {
        "type": "string",
        "description": "Filter: 'naming', 'imports', 'errors', 'dependencies', 'testing', 'visibility', 'functions', 'git_health', or 'all' (default 'all')",
        "default": "all"
      },
      "strength": {
        "type": "string",
        "description": "Minimum strength: 'convention', 'trend', 'mixed', or 'all' (default 'all')",
        "default": "all"
      },
      "focus": {
        "type": "string",
        "description": "Path prefix — recompute stats scoped to this directory"
      }
    }
  }
}
```

Returns the `ConventionProfile` serialized to JSON, filtered by params. When `focus` is provided, stats are recomputed by iterating the per-file contribution maps and filtering by path prefix. This is O(n) over files in the subtree, not O(1) — acceptable since `cxpak_conventions` is a low-frequency exploratory tool, not called in the hot path. Conventions may differ between `src/api/` and `src/services/`.

## Feature 4: DNA Section in `auto_context`

### Pipeline Change

`auto_context` gains step 0 — before all other steps:

```
0. Convention DNA (~1000 tokens, always included, never degraded)
    ↓
1. Query expansion
    ↓
2-10. (existing pipeline)
```

### Budget Accounting

DNA section is deducted from the token budget BEFORE fill-then-overflow allocation.

**Return type:** Add `pub dna: String` field to `AutoContextResult`. The DNA markdown is stored here and serialized as part of the MCP response. The MCP handler prepends it to the response content. `PackedSections` is not modified — DNA is a peer of sections, not a section itself. With a 50k budget: ~1000 goes to DNA, ~49000 goes to code/tests/schema/blast/API sections.

The DNA section is NEVER degraded or dropped regardless of budget pressure. It applies globally to all generated code, making it the highest signal-per-token content in the context.

**Minimum budget:** If the requested budget is less than 2000 tokens, DNA is skipped entirely (the budget is too small for DNA to be useful alongside actual code). Between 2000-5000 tokens, a compact ~300 token DNA summary is used (top 3 conventions only). Above 5000, the full ~1000 token DNA is included.

### Rendering

`render_dna_section(profile: &ConventionProfile) -> String`

Produces ~800-1000 tokens of compact markdown. Contents:
- All Convention-strength patterns (≥90%) with counts and percentages
- All Trend-strength patterns (70-89%) with counts
- Top 5 git health entries (highest 30-day churn)
- Revert history (anti-patterns the project has explicitly rejected)

Ordered by: conventions first (sorted by percentage desc), then trends, then git health.

Example output:

```markdown

### Naming (convention)
- Functions: snake_case (347/350, 99.1%) — 3 exceptions: FFI bindings
- Types: PascalCase (89/89, 100%)
- Files: snake_case, match module name (98%)
- Constants: SCREAMING_SNAKE_CASE (100%)

### Error Handling (convention)
- Return type: Result<T, AppError> (147/150, 98%) — never unwrap() in src/
- Propagation: ? operator (143/147, 97%)
- Error types: one per module, all implement From<underlying> (100%)
- Reverted: 3 commits removed unwrap() usage — confirmed anti-pattern

### Imports (convention)
- Style: absolute paths (96%), never relative
- Grouping: std → external → crate:: → super::, blank line between groups
- Re-exports: each mod.rs re-exports public types (89%)

### Architecture (convention)
- Layering: api/ → services/ → repository/ (0 reverse dependencies)
- Database: only repository/ imports sqlx (100%)
- No circular deps between top-level modules

### Visibility (trend, 74%)
- 34% public / 66% private — default to private
- Doc comments on 78% of public APIs

### Functions (trend, 82%)
- Average length: 23 lines (median 18)
- src/api/ handlers: avg 15 lines — thin handlers
- src/services/: avg 30 lines — denser business logic

### Testing (convention)
- Coverage: 1:1 test files for src/api/ (100%), src/services/ (92%)
- Style: real DB for integration, builders for unit — 0 mock libraries
- Names: test_{fn}_{scenario}_{expected} (87%)
- Density: avg 4.2 tests per public function

### Git Health (30d / 180d)
- Hot: src/index/mod.rs (47 / 52) — active development
- Hot: src/commands/serve.rs (31 / 89) — growing rapidly
- Stable: src/output/ (0 / 3) — mature
- Bug density: src/api/ 0.31/file, src/output/ 0.02/file
- Reverts: 3× unwrap() removal, 1× circular dep fix
```

### The Contract

The DNA section creates a contract: "I told you the rules, backed by evidence." When the LLM then calls `cxpak_verify {}`, violations reference the same conventions. The LLM can reason: "I was told 99.1% of functions use snake_case — I should fix this PascalCase function."

## Design Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Build time extraction | Cached on CodebaseIndex, incremental updates | Feeds auto_context which is the primary tool; must be precomputed |
| Typed categories + dynamic entries | Struct per category with Vec<PatternObservation> | Type safety for known patterns, extensibility for new ones |
| Verify against cached profile | Not live re-scan | File watcher keeps profile fresh; consistent with other cached data |
| Tiered strength labels | Convention ≥90%, Trend 70-89%, Mixed 50-69% | Maps cleanly to verify severity; gives LLM context on rule strictness |
| Two git windows (30d + 180d) | Not single window | Two dimensions reveal trend: hot/stabilizing/chronic/cold |
| Report what IS not SHOULD BE | Evidence-based, never prescriptive | Deterministic, defensible, no opinions; LLM draws conclusions |
| Only verify changed lines | Skip pre-existing debt | Prevents overwhelming the LLM with violations it didn't introduce |
| Deterministic suggestions only | null when judgment needed | Never generate a wrong suggestion; honest about limits |
| ~1000 token DNA always included | Never degraded regardless of budget | Highest signal-per-token — applies to ALL generated code, 2% budget for 10x quality |
| DNA deducted before fill-then-overflow | Not part of the section priority system | Conventions are not a "section" — they're a universal constraint |
| 13 MCP tools total | Add verify + conventions | verify is the high-value tool; conventions is for deep exploration |
