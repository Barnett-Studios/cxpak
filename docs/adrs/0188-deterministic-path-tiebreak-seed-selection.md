# ADR-0188: Deterministic path-asc tiebreak in seed selection, noise dedup, and briefing pack

## Context

Immediately after the ADR-0187 flip to Active (RRF) ranking, the bench gate caught
that `auto_context` file selection is **non-deterministic across processes** at a
tight token budget. Same code (HEAD `8d76d46`), same cached repos, only the process
differs: `regenerate_baseline` (process A) measured recall@8k **0.6444**, a fresh
`recall_gate_holds` run (process B) measured **0.6111**. recall@32k was stable
(0.6444 both) — i.e. the SELECTED FILE SET for a PR varies run-to-run only where the
budget cut lands on tied scores.

Root cause: `select_seeds` (`src/relevance/seed.rs`) collected `result_map.into_values()`
— HashMap iteration order, randomized per process by SipHash — and sorted by score
alone (a STABLE sort with NO secondary key). Equal-scored files therefore kept the
random HashMap order, and `truncate(limit)` plus the downstream budget pack cut a
different file set per process. The same tiebreak-less float sort existed in the
briefing pack (`src/auto_context/briefing.rs`), which sets fill-then-overflow packing
order. Two secondary leaks in the same pipeline: the seed fan-out loop iterated
`result_map.keys()` in HashMap order while MUTATING the map (a neighbor that is itself
a scored seed can have its score RAISED mid-loop, and that raised score becomes the
basis for its own fan-out — so processing order changed derived scores); and the
noise dedup (`src/auto_context/noise.rs`) resolved a full score+PageRank tie by
keeping the earlier Vec index, which upstream may derive from HashMap order.

Active did not INTRODUCE the leak — it EXPOSED a latent one. Inert's weighted-sum
scores are spread out, so ties at the budget boundary are rare. RRF's `(K+1)/(K+n)`
worst-file floor degenerates on large repos (ADR-0187): many files land on the same
floor value, so tie-broken selection now dominates the budget-boundary decision. This
is a human decision because determinism is the project's core promise; the fix must
pin a specific, documented total order rather than let the runtime pick one.

## Options considered

- **Option A — path-ascending secondary tiebreak on every selection-path float sort
  (chosen):** make each score sort a strict total order `(score desc, path asc)`, sort
  the fan-out iteration keys, and give the dedup survivor a `(score, PageRank, path)`
  total order. Cheap, local, matches the existing RRF path-asc convention (rrf.rs,
  reranker.rs). Con: the deterministic winner of a tie MAY differ from the old
  accidental winner (a one-time, intended selection change, not a regression).
- **Option B — sort `all_scored` once upstream and rely on stable sorts downstream:**
  fewer edit sites. Con: fragile — any intervening HashMap round-trip (select_seeds
  rebuilds a HashMap from its input) re-randomizes order, so an upstream sort does not
  survive to the budget cut. Does not actually close the leak.
- **Option C — switch the maps to a deterministic-iteration container (BTreeMap /
  IndexMap):** removes HashMap randomness at the source. Con: broader blast radius,
  changes hot-path allocation characteristics, and still needs an explicit tiebreak in
  the score sorts anyway (BTreeMap key is path, not score). New dep for IndexMap.

## Decision

Option A. Add a deterministic path-ascending secondary key to the two tiebreak-less
float sorts (`seed.rs` final sort, `briefing.rs` target pack), sort the seed fan-out
iteration keys before the mutating loop, and extend the noise dedup keep-decision to a
full `(relevance score desc, PageRank desc, path asc)` total order so the removed file
is a pure function of the inputs. The bench baseline is regenerated by the controller
AFTER this fix, once selection is reproducible.

## Consequences

### Positive
- `auto_context` file selection is reproducible across processes: identical recall on
  repeated bench runs; determinism (the core promise) is restored under Active.
- The four order sources are pinned to one documented total order that matches the
  existing RRF/reranker path-asc convention, so the whole ranking path is consistent.
- Teeth regression tests assert the observable guarantee (`select_seeds` output is
  `(score desc, path asc)` on tied scores; dedup ties resolve to the smaller path
  regardless of input Vec order), so a future tiebreak removal fails a unit test.

### Negative
- On a tie, the deterministic winner may differ from the old accidental winner — a
  one-time selection shift. The bench baseline is regenerated to reflect it. No
  auto_context-asserting test moved (verified); the visual `spa_output_matches_golden_fixture`
  is on the PageRank/visual path and stays byte-identical.

### Neutral
- No new dependencies; no LLM; sorts stay `O(n log n)`. HashMaps are retained (the
  tiebreak, not the container, provides determinism).

## Revisit if

- The seed/noise/briefing maps are migrated to a deterministic-iteration container —
  then the explicit score-sort tiebreaks are still required but the fan-out key sort
  becomes redundant.
- A future signal introduces genuine NaN scores — `total_cmp` already orders them
  deterministically, but the semantics should be reviewed.
- The bench gate ever again reports differing recall across processes on identical
  inputs — indicates a new order leak elsewhere in the selection path.
